Módulo 6: Infraestrutura de Segurança de Rede

Explicar como os dispositivos e serviços são usados para aumentar a segurança da rede.

6.0.1 Por que devo fazer esse Módulo?

Com as muitas ameaças à segurança da rede, como as redes podem ser projetadas para proteger recursos de dados e garantir que os serviços de rede sejam fornecidos conforme necessário? A infra-estrutura de segurança de rede define a forma como os dispositivos são conectados entre si para obter comunicações seguras de ponta a ponta.

Assim como existem muitos tamanhos de redes, há também muitas maneiras de construir uma infraestrutura de rede segura. No entanto, existem alguns projetos padrão que o setor de rede recomenda alcançar redes que estão disponíveis e seguras. Este capítulo abrange o funcionamento básico das infra-estruturas de rede, os vários dispositivos de segurança da rede e os serviços de segurança utilizados para monitorizar e manter a transmissão segura e eficiente de dados.

6.1.2 Firewalls

Operação de firewall

Um firewall é um sistema ou grupo de sistemas que aplica uma política de controle de acesso entre redes.

Animações - Regras de Firewall

Permitir tráfego de qualquer endereço externo para o servidor Web
Permitir tráfego no servidor FTP
Permitir tráfego para o servidor SMTP
Permitir tráfego no servidor IMAP interno
Negar tráfego de entrada com endereços de rede que correspondam a IPs internos
Negar tráfego de entrada de endereços externos para o servidor
Negar tráfego de entrada de ICMP echo requests
Negar consultas de entrada ao MS Active Directory
Negar tráfego de entrada para consultas no servidor MS SQL
Negar transmissões do MS Domain Local

Propriedades comuns do Firewall

Todos os firewalls compartilham algumas propriedades comuns:

Resistência a ataques

Os firewalls são resistentes a ataques de rede.

Ponto único de trânsito

Firewalls são o único ponto de trânsito entre redes corporativas internas e redes externas.

Política de controle

Firewalls reforçam a política de controle de acesso.

Benefícios do firewall

Existem vários benefícios do uso de um firewall em uma rede:

  • Eles impedem a exposição de hosts, recursos e aplicações sensíveis a usuários não confiáveis.
  • Eles sanitizam o fluxo do protocolo, o que impede a exploração de falhas no protocolo.
  • Eles bloqueiam dados maliciosos de servidores e clientes.
  • Eles reduzem a complexidade do gerenciamento de segurança descarregando a maior parte do controle de acesso à rede para alguns firewalls na rede.

Limitações do Firewall

Os firewalls também têm algumas limitações:

  • Um firewall mal configurado pode ter sérias conseqüências para a rede, como se tornar um único ponto de falha.
  • Os dados de muitos aplicativos não podem ser transmitidos por firewalls com segurança.
  • Os usuários podem procurar proativamente maneiras de contornar o firewall para receber material bloqueado, o que expõe a rede a possíveis ataques.
  • O desempenho da rede pode diminuir.
  • O tráfego não autorizado pode ser encapsulado ou escondido como tráfego legítimo através do firewall.

6.1.3 Arquiteturas comuns de segurança

O design do firewall é principalmente sobre interfaces de dispositivo que permitem ou negam tráfego com base na origem, no destino e no tipo de tráfego. Alguns designs são tão simples quanto designar uma rede externa e uma rede interna, que são determinados por duas interfaces em um firewall.

Privado e Público Zona Desmilitarizada Firewalls de política baseados em zona

Privado e Público

Como mostrado na figura, a rede pública (ou rede externa) não é confiável e a rede privada (ou rede interna) é confiável.

Normalmente, um firewall com duas interfaces é configurado da seguinte forma:

  • O tráfego proveniente da rede privada é permitido e inspecionado à medida que viaja em direção à rede pública. É permitido o tráfego inspecionado que retorna da rede pública e associado ao tráfego originado da rede privada.
  • O tráfego originado da rede pública e que viaja para a rede privada geralmente é bloqueado.

Zona Desmilitarizada (DMZ)

Uma zona desmilitarizada (DMZ) é um projeto de firewall onde normalmente há uma interface interna conectada à rede privada, uma interface externa conectada à rede pública e uma interface DMZ.

  • Privada → Pública/DMZ: Tráfego inspecionado com pouca ou nenhuma restrição. Retorno permitido.
  • DMZ → Privada: Geralmente bloqueado.
  • DMZ → Pública: Permitido seletivamente com base nos requisitos de serviço.
  • Pública → DMZ: Seletivamente permitido e inspecionado (email, DNS, HTTP, HTTPS). Retorno permitido.
  • Pública → Privada: Bloqueado.

Firewalls de política baseados em zona (ZPF)

Os firewalls de política baseados em zona (ZPFs) usam o conceito de zonas para fornecer flexibilidade adicional. Uma zona é um grupo de uma ou mais interfaces que têm funções ou recursos semelhantes. As zonas ajudam a especificar onde uma regra ou política de firewall do Cisco IOS deve ser aplicada.

LAN 1

Membro da mesma zona

LAN 2

Membro da mesma zona

Na figura, as políticas de segurança para LAN 1 e LAN 2 são semelhantes e podem ser agrupadas em uma zona para configurações de firewall. Por padrão, o tráfego entre interfaces na mesma zona não está sujeito a nenhuma política e passa livremente. No entanto, todo o tráfego de zona para zona está bloqueado. Para permitir o tráfego entre as zonas, uma política que permite ou inspeciona o tráfego deve ser configurada.

Zona própria do roteador: A única exceção a este padrão "deny any" é a zona própria do roteador. A zona auto é o próprio roteador e inclui todos os endereços IP da interface do roteador. As configurações de política que incluem a zona automática aplicar-se-iam ao tráfego destinado e proveniente do roteador. Por padrão, não há nenhuma política para esse tipo de tráfego. O tráfego que deve ser considerado ao projetar uma política para a auto zona inclui o tráfego de plano de gerenciamento e plano de controle, como SSH, SNMP e protocolos de roteamento.

6.1.4

Descrições de tipos de firewall

É importante entender os diferentes tipos de firewalls e suas capacidades específicas para que o firewall correto seja usado para cada situação.

Firewall de filtragem de pacotes
(sem estado)		 Firewall com monitoração de estado Firewall de gateway de aplicativo Firewall de próxima geração

Filtragem de Pacotes

Sem estado

Geralmente fazem parte de um firewall de roteador, que permite ou nega tráfego com base nas informações da Camada 3 e Camada 4.

São firewalls sem estado que usam uma simples pesquisa de tabela de políticas que filtra o tráfego com base em critérios específicos.

Exemplo:

Servidores SMTP escutam a porta 25. Um admin pode configurar o firewall para bloquear a porta 25 de uma estação de trabalho específica para impedir que ele transmita um vírus de e-mail.

Monitoração de Estado

Stateful

São as tecnologias de firewall mais versáteis e mais comuns em uso.

Fornecem filtragem de pacotes stateful usando informações de conexão mantidas em uma tabela de estado.

Arquitetura classificada na camada de rede. Analisa o tráfego na camada 4 da OSI e na camada 5.

  • Mantém tabela de conexões ativas
  • Mais seguro que filtragem simples

Gateway de Aplicativo

Proxy

Filtra informações nas camadas 3, 4, 5 e 7 do modelo OSI.

A maior parte do controle e filtragem é feita em software.

Quando um cliente precisa acessar um servidor remoto, ele se conecta a um servidor proxy.

O servidor proxy se conecta ao servidor remoto em nome do cliente.

O servidor só vê uma conexão do servidor proxy.

Próxima Geração

NGFW

Vão além dos firewalls de estado, fornecendo:

  • Prevenção de intrusão integrada
  • Reconhecimento e controle de aplicativos
  • Bloquear aplicativos arriscados
  • Caminhos de atualização para futuros feeds
  • Técnicas para ameaças em evolução

Outros métodos de implementação de firewalls

Firewalls baseados em host

Um PC ou servidor com software de firewall em execução nele. Protege o dispositivo individualmente.

Firewall transparente

Filtra o tráfego IP entre um par de interfaces em ponte. Opera sem precisar de configuração de IP nos hosts.

Firewall híbrido

Uma combinação dos vários tipos de firewall. Ex: firewall de inspeção de aplicações combina um firewall com estado com um firewall de gateway de aplicativo.

Nota Cisco: A escolha do firewall depende do cenário de rede, requisitos de segurança e desempenho necessário. Redes modernas frequentemente utilizam uma combinação de múltiplos tipos (defesa em profundidade).
6.2.1

Controle de tráfego com ACLs

Access Control Lists (ACLs) são o mecanismo fundamental para permitir ou negar tráfego em roteadores Cisco

O que são ACLs?

ACLs são listas sequenciais de instruções "permit" ou "deny" aplicadas em interfaces de roteadores/switches para controlar tráfego:

  • ACLs padrão (1-99): Filtram apenas por IP de origem
  • ACLs estendidas (100-199): Filtram por IP origem/destino, porta, protocolo
  • ACLs nomeadas: Mais fáceis de gerenciar
📌 Exemplo ACL estendida 
access-list 100 permit tcp 192.168.1.0 0.0.0.255 any eq 80
access-list 100 deny ip any any
interface g0/0
 ip access-group 100 in

Funcionamento prático

As ACLs são avaliadas em ordem sequencial - na primeira correspondência, a ação é executada:

1 Pacote chega na interface
2 ACL existe? → Não: permite
3 Sim: compara com linha 1
4 Match? → Executa permit/deny
5 Não: próxima linha
6 Fim da lista? → Deny implícito
Importante: Toda ACL tem um "deny any any" implícito no final!
6.2.2

Serviços de monitoramento de rede

SNMP, NetFlow, Syslog e NTP trabalham juntos para manter a visibilidade e segurança da infraestrutura

SNMP (Simple Network Management Protocol)

Protocolo para coleta de informações e gerenciamento de dispositivos de rede:

  • SNMPv1/v2c: Comunidade como senha (fraca)
  • SNMPv3: Autenticação e criptografia
  • MIB (Management Information Base): Banco de dados com informações do dispositivo
snmp-server community public RO
snmp-server community private RW
snmp-server location "Data Center SP"
snmp-server contact "admin@empresa.com"

NetFlow

Tecnologia Cisco para coleta de metadados do tráfego IP:

  • Quem? (IP origem/destino)
  • O quê? (Portas, protocolo)
  • Quando? (Timestamp)
  • Quanto? (Bytes/pacotes)

Aplicação: Detecção de anomalias, análise de largura de banda, investigação forense

Espelhamento de portas (SPAN)

Cópia do tráfego de uma porta/plan para análise:

  • IDS/IPS: Analisar tráfego em tempo real
  • Wireshark: Troubleshooting
  • Registro forense: Evidências
monitor session 1 source interface g0/1 both
monitor session 1 destination interface g0/24

Syslog e NTP

Syslog: Centralização de logs de todos os dispositivos

  • 0 - Emergências
  • 1-2: Alertas/Crítico
  • 3-4: Erro/Aviso
  • 5-7: Notificação/Informação/Debug

NTP (Network Time Protocol): Sincronização de tempo CRÍTICA para:

  • Correlacionar logs entre dispositivos
  • Validade de certificados digitais
  • Auditoria e conformidade
6.2.3

Autenticação, Autorização e Controle Remoto

AAA (Authentication, Authorization, Accounting) e VPN garantem acesso seguro aos recursos

Servidores AAA

Framework de segurança para controle de acesso:

🔐 Authentication: Quem é você? (Login/senha, tokens, biometria)
📋 Authorization: O que você pode fazer? (Permissões, níveis de acesso)
📊 Accounting: O que você fez? (Logs, auditoria, billing)
Protocolos:
  • RADIUS (UDP 1812): Cisco, wireless, switches - apenas criptografa senha
  • TACACS+ (TCP 49): Cisco - criptografia completa, separação AAA
aaa new-model
aaa authentication login default group tacacs+ local
aaa authorization exec default group tacacs+ local
tacacs-server host 192.168.1.100 key CISCO123

VPN (Virtual Private Network)

Conexão segura através de redes públicas:

Site-to-Site VPN

Conecta escritórios inteiros via Internet (IPsec/GRE)

Remote Access VPN

Usuários remotos conectam-se à rede corporativa (SSL/AnyConnect)

Componentes IPsec:
  • IKE (Internet Key Exchange): Negociação de chaves (UDP 500)
  • ESP (Encapsulating Security Payload): Criptografia dos dados
  • AH (Authentication Header): Autenticação (sem criptografia)
Caso real: Funcionário em casa → Internet → VPN → Rede corporativa (acesso como se estivesse no escritório)

🎯 Laboratório Mental: Implementando segurança na "EmpresaXYZ"

Aplicação integrada de TODOS os conceitos do módulo em um cenário empresarial real

📋 Cenário inicial

A EmpresaXYZ tem:

  • Matriz em SP (200 funcionários) + Filial no RJ (50 funcionários)
  • Servidores Web, Email e DNS públicos
  • Funcionários remotos precisam acessar sistemas internos
  • PCI DSS (segurança para cartões) e LGPD

Firewalls

  • Perímetro: NGFW (Firepower) com IPS integrado
  • DMZ: Servidores Web/Email/DNS isolados
  • ZPF: Zonas: Trust (funcionários), DMZ (servidores), Untrust (Internet)
  • Regras: Permitido apenas HTTP/HTTPS para DMZ, SSH apenas da rede interna

ACLs

  • ACL 110: Permitir HTTP da Internet para Web server (192.168.2.10)
  • ACL 120: Bloquear acesso à rede interna (10.0.0.0/8) vindo da DMZ
  • ACL 130: Permitir apenas SSH da rede admin (10.1.1.0/24) para roteadores

Monitoramento

  • SNMPv3: Monitoramento de CPU/memória dos equipamentos
  • NetFlow: Análise de tráfego - detectou botnet tentando C&C
  • Syslog: Logs centralizados no servidor Graylog
  • NTP: Todos dispositivos sincronizados com pool.ntp.br

AAA e VPN

  • TACACS+: Administradores autenticados com 2FA (Google Authenticator)
  • RADIUS: Autenticação 802.1X para WiFi corporativo
  • VPN IPsec: Túnel site-to-site SP-RJ (circuito dedicado virtual)
  • SSL VPN (AnyConnect): Home office acessa sistemas internos
