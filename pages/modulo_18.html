Módulo 18: Criptografia e Confidencialidade de Dados

Fundamentos de criptografia simétrica e assimétrica para segurança de dados em redes Cisco.

18.1.1 Confidencialidade de Dados

Existem duas classes de criptografia usadas para fornecer confidencialidade de dados: assimétrica e simétrica. Essas duas classes diferem na maneira como usam as chaves.

Criptografia Simétrica

  • Usa a mesma chave para criptografar e descriptografar
  • Comprimentos de chave: 40 - 256 bits
  • Mais rápida que a assimétrica
  • Usada para dados em massa (VPN)

Criptografia Assimétrica

  • Usa chaves diferentes para cifrar/decifrar
  • Comprimentos de chave: 512 - 4096 bits
  • Mais lenta (alto custo computacional)
  • Usada para transações rápidas (HTTPS)
Nota: DES é um algoritmo legado e não deve ser usado. 3DES deve ser evitado quando possível.

18.1.2 Criptografia Simétrica

Algoritmos simétricos usam a mesma chave pré-compartilhada para criptografar e descriptografar dados. Uma chave pré-compartilhada (chave secreta) é conhecida pelo remetente e receptor antes de qualquer comunicação criptografada.

Exemplo: Alice e Bob

Chave Pré-Compartilhada → Mesmo cadeado 🔒

Alice
Mensagem → Tranca
Bob
Destranca → Mensagem 
# Conceito de Criptografia Simétrica
Chave_Secreta = "precompartilhada"
Mensagem_Original = "Confidencial"
Mensagem_Cifrada = encrypt(Mensagem_Original, Chave_Secreta)
Mensagem_Decifrada = decrypt(Mensagem_Cifrada, Chave_Secreta)
# Mensagem_Decifrada == Mensagem_Original ✅

18.1.3 Algoritmos de Criptografia Simétrica

Hoje, algoritmos de criptografia simétrica são comumente usados com tráfego VPN por usarem menos recursos de CPU. Para garantir segurança, use comprimento mínimo de chave de 128 bits.

Algoritmo Descrição
DES Algoritmo legado. Chave curta, inseguro para uso atual.
3DES Substituto do DES, repetido 3 vezes. A ser aposentado em 2023.
AES Padrão recomendado. Chaves de 128, 192 ou 256 bits.
SEAL Cifra de fluxo, 160 bits, baixo impacto na CPU.
RC4 Vulnerabilidades conhecidas. Não utilizar.
Cifra de Bloco

Criptografa dados em blocos fixos (ex: 128 bits). AES opera em blocos.

Cifra de Fluxo

Criptografa um byte/bit por vez. Exemplos: RC4, A5 (GSM).

01010010 → bit a bit

18.1.4 Criptografia Assimétrica - Confidencialidade

Algoritmos assimétricos fornecem confidencialidade sem pré-compartilhar senha. O processo é iniciado com a chave pública.

Fórmula: Chave pública (criptografar) + Chave privada (descriptografar) = Confidencialidade
Processo de Troca Segura (Bob e Alice)
1. Alice adquire chave pública de Bob

Alice solicita e obtém a chave pública de Bob.

Alice → Bob: "Você poderia me informar sua chave pública?"
Bob → Alice: "Aqui está minha chave pública."
2. Alice usa a chave pública

Alice criptografa a mensagem usando a chave pública de Bob.

Mensagem + Chave_Pública_Bob → Texto Criptografado
3. Bob descriptografa com chave privada

Bob usa sua chave privada (única) para descriptografar a mensagem. 

# Apenas Bob pode descriptografar
if chave_privada_bob == correta:
    mensagem_original = decrypt(texto_criptografado)

🔐 Segurança Garantida: Como apenas Bob tem a chave privada, a confidencialidade é alcançada.

⚠️ Se a chave privada for comprometida, um novo par de chaves deve ser gerado.

18.1.5 Criptografia Assimétrica - Autenticação

A autenticação é iniciada quando o processo de criptografia usa a chave privada.

Fórmula: Chave privada (criptografar) + Chave pública (descriptografar) = Autenticação
Processo de Autenticação (Bob e Alice)
1. Alice usa sua chave privada

Alice criptografa uma mensagem usando sua chave privada e envia para Bob. 

Alice (Chave_Privada_Alice) → Criptografa → Mensagem_Autenticada → Bob
2. Bob solicita a chave pública de Alice

Bob precisa verificar se a mensagem realmente veio de Alice.

Bob → Alice: "Posso pegar sua chave pública, por favor?"
Alice → Bob: "Aqui está minha chave pública."
3. Bob descriptografa usando a chave pública

Se a descriptografia funcionar, a mensagem foi criptografada pela chave privada de Alice. 

# Verificação de autenticidade
if decrypt(mensagem_recebida, chave_publica_alice) == sucesso:
    print("✅ Mensagem autenticada: vem de Alice")

🆔 Autenticidade: Como apenas Alice tem sua chave privada, somente ela poderia ter criptografado a mensagem, provando sua identidade.

📊 Algoritmos de Criptografia Assimétrica

Algoritmo Comprimento da Chave Descrição

DSS / DSA 512 - 1024 bits Algoritmo para assinaturas digitais. Mais lento para verificação (10-40x).
RSA 512 - 2048 bits Baseado na dificuldade de fatorar números muito grandes. Amplamente usado em e-commerce.

18.1.6 Criptografia Assimétrica - Integridade

Combinar os dois processos de criptografia assimétrica (confidencialidade e autenticação) fornece confidencialidade, autenticação e integridade da mensagem.

Objetivo: Garantir que apenas Bob leia a mensagem, que Alice seja a remetente e que a mensagem não foi modificada.
Passo 1: Confidencialidade

Alice usa a chave pública de Bob para criptografar a mensagem original.

Mensagem + Chave_Pública_Bob → Texto Criptografado

✅ Apenas Bob poderá descriptografar com sua chave privada.

Passo 2: Autenticação e Integridade

Alice cria um hash da mensagem e criptografa o hash usando sua chave privada.

Hash(Mensagem) + Chave_Privada_Alice → Hash Criptografado

✅ Isso prova a identidade de Alice e garante integridade.

Passo 3: Bob descriptografa o hash

Bob usa a chave pública de Alice para descriptografar o hash recebido.

Hash_Criptografado + Chave_Pública_Alice → Hash_Recebido

✅ Se funcionar, a mensagem veio de Alice (autenticação).

Passo 4: Bob descriptografa a mensagem

Bob usa sua chave privada para descriptografar a mensagem original.

Texto_Criptografado + Chave_Privada_Bob → Mensagem_Original

✅ Confidencialidade garantida.

Verificação Final de Integridade

Bob calcula o hash da mensagem descriptografada e compara com o hash recebido: 

# Processo de verificação no lado de Bob
hash_calculado = hash(Mensagem_Original)
hash_recebido = decrypt(Hash_Criptografado, Chave_Pública_Alice)

if hash_calculado == hash_recebido:
    print("✅ Integridade confirmada - Mensagem não foi modificada")
    print("✅ Autenticidade confirmada - Mensagem veio de Alice")
else:
    print("❌ Mensagem comprometida ou remetente não autenticado")
Resumo: Este processo combinado garante os três pilares da segurança:
  • Confidencialidade - Chave pública de Bob
  • Autenticação - Chave privada de Alice
  • Integridade - Hash criptografado verificado

18.1.7 Diffie-Hellman (DH)

Diffie-Hellman (DH) é um algoritmo matemático assimétrico que permite que dois computadores gerem um segredo compartilhado idêntico sem terem se comunicado antes. A nova chave compartilhada nunca é realmente trocada entre o remetente e o destinatário, mas ambas as partes a conhecem.

Aplicações comuns:
  • VPN IPsec (trocas de chaves IKE)
  • SSH (Secure Shell)
  • TLS/SSL (handshake)
Analogia das Cores (Simplificação do DH)

Para entender o DH, vamos usar uma analogia com cores (no mundo real, são números extremamente grandes):

Passo 1: Acordo Público

Alice e Bob concordam com uma cor comum arbitrária que não precisa ser mantida em segredo.

→ Cor Combinada: AMARELO
Passo 2: Cores Secretas (Chaves Privadas)

Cada um seleciona uma cor secreta que nunca será compartilhada.

Alice
🔴 Vermelho

Bob
🔵 Azul

Passo 3: Mistura e Troca Pública

Cada um mistura a cor comum com sua cor secreta para produzir uma cor pública.

🟡 Amarelo + 🔴 Vermelho = 🟠 Laranja

Alice envia LARANJA para Bob

🟡 Amarelo + 🔵 Azul = 🟢 Verde

Bob envia VERDE para Alice

Passo 4: Mistura Final (Segredo Compartilhado)

Cada um mistura a cor recebida com sua cor secreta original.

🟢 Verde (recebido) + 🔴 Vermelho (secreto) = 🟤 MARROM

Alice calcula

🟠 Laranja (recebido) + 🔵 Azul (secreto) = 🟤 MARROM

Bob calcula

Resultado: Ambos obtêm a mesma cor (MARROM) - este é o segredo compartilhado!
Grupos Diffie-Hellman no Cisco IOS

No mundo real, DH usa números extremamente grandes. O Cisco IOS suporta diferentes grupos DH, onde números maiores significam mais segurança, porém maior tempo de processamento.

Grupo DH Tamanho da Chave (bits) Nível de Segurança
1 768 bits 🔓 Baixo (obsoleto)
2 1024 bits ⚠️ Médio-Baixo (evitar)
5 1536 bits 📈 Médio
14 2048 bits ✅ Bom (recomendado)
15 3072 bits 🔒 Muito Bom
16 4096 bits 🛡️ Excelente (alta segurança)
19, 20, 24 Curva Elíptica (ECC) 🔐 Eficiente e seguro
Comandos Cisco para Configurar DH 
! Configurar política IKE com grupo DH
Router(config)# crypto isakmp policy 10
Router(config-isakmp)# encryption aes 256
Router(config-isakmp)# group 14   ! Recomendado: 2048 bits
Router(config-isakmp)# exit

! Verificar grupos DH suportados
Router# show crypto isakmp policy

! Verificar transform sets com DH
Router# show crypto ipsec transform-set
Por que usar DH + Algoritmos Simétricos?

Sistemas de chave assimétrica são lentos para criptografia em massa. Por isso, usamos o DH para criar uma chave compartilhada segura, e então utilizamos algoritmos simétricos (como AES) para criptografar o tráfego real - combinando segurança e desempenho!

18.2.2 Técnicas de Mascaramento de Dados

O mascaramento de dados protege informações sensíveis, substituindo-as por uma versão pública que parece e age como o original, permitindo que processos organizacionais usem dados não confidenciais sem alterações nos sistemas de suporte.

Objetivo Principal

Limitar a propagação de dados sensíveis dentro de sistemas de TI, distribuindo conjuntos de dados substitutos para teste e análise, sem expor informações reais.

Técnicas de Mascaramento
Substituição

Troca os dados por valores que parecem autênticos, tornando os registros anônimos, mas mantendo a aparência realista.

João Silva → Cliente #12345
1234-5678 → ****-****
Embaralhamento

Origina um conjunto de substituição da mesma coluna de dados, ideal para informações financeiras em bancos de dados de teste.

Original: [R$100, R$200, R$300]
Embaralhado: [R$300, R$100, R$200]
Anulação

Aplica um valor nulo a um campo específico, impedindo totalmente a visibilidade dos dados.

Campo: CPF
Valor Original: 123.456.789-00
Após Anulação: NULL
Benefício: Todas as técnicas garantem que os dados permaneçam significativos para processos de teste, mas sejam alterados o suficiente para proteger informações sensíveis.

18.2.3 Esteganografia

A esteganografia é a arte de esconder dados (a mensagem secreta) dentro de outro arquivo, como uma imagem, áudio, vídeo ou arquivo de texto.

Vantagem sobre a criptografia: A mensagem secreta não atrai atenção especial. Ao visualizar o arquivo normalmente, ninguém saberá que ele contém uma mensagem oculta.

🔍 Técnicas de Esteganografia

Bits Menos Significativos (LSB)

A abordagem mais comum para integrar dados em imagens é o uso dos Bits Menos Significativos (LSB - Least Significant Bits).

Como funciona o LSB:
  • Um pixel é a unidade básica de cor em uma imagem
  • Cada pixel é uma mistura de RGB (Vermelho, Verde, Azul)
  • Três bytes especificam a cor de um pixel (1 byte por cor)
  • Sistema de 24 bits usa todos os três bytes
  • LSB usa um bit de cada componente RGB → 3 bits por pixel

Olho humano não percebe alterações nos bits menos significativos!

Em média, apenas metade dos bits de uma imagem precisa ser alterada para ocultar uma mensagem eficazmente.

Exemplo Prático - Ocultando a Letra "T" (ASCII 84 = 01010100)
Pixel RGB Original RGB Modificado (com dados ocultos)
Pixel 1 00100110 11101001 11010000 00100111 11101001 11010000
Pixel 2 00100111 11001000 11101001 00100111 11001001 11101001
Pixel 3 11001000 00100110 11001001 11001000 00100110 11001001

⚠️ Apenas dois bits foram alterados - imperceptível visualmente!

👥 Esteganografia Social

A esteganografia social oculta informações à vista de todos, criando uma mensagem que pode ser lida de forma específica apenas pela pessoa que recebe a mensagem.

Uso em Redes Sociais

Adolescentes usam essa tática para se comunicar com amigos próximos, evitando que outras pessoas (como pais) percebam o real significado.

Exemplo: "Ir ao cinema" 🎬 pode significar "Ir à praia" 🏖️
Uso em Países com Censura

Pessoas em países que censuram a mídia usam esteganografia social digitando palavras incorretamente ou fazendo referências imprecisas para enviar duas mensagens simultaneamente:

  • Mensagem aparente - para censores
  • Mensagem secreta - para destinatário real
Exemplo de Comunicação Dual

"O clima hoje está ótimo para um piquenique no parque."

Mensagem Aparente

"Encontro hoje às 15h no local combinado."

Mensagem Secreta (apenas para quem sabe decodificar)

🕵️ Detecção - Estegoanálise

Estegoanálise é a prática de descobrir a existência de informações ocultas. O objetivo é detectar e, quando possível, extrair as mensagens secretas.

Métodos de Detecção
Análise de Disco

Utilidades de análise podem relatar informações ocultas em clusters não utilizados ou áreas reservadas de dispositivos de armazenamento.

# Verificar clusters não alocados
fsck -n /dev/sda1
Análise de Rede

Filtros podem capturar pacotes que contêm informações ocultas nos cabeçalhos ou campos não utilizados dos protocolos.

# Capturar pacotes suspeitos
tcpdump -i eth0 -w capture.pcap
Análise Visual

Comparando uma imagem original com a estego-imagem, analistas podem buscar padrões de repetição visual ou anomalias estatísticas.

Assinaturas de esteganografia
Desafio: A estegoanálise é uma área em constante evolução, pois novas técnicas de ocultação surgem continuamente para evitar a detecção.
Indicadores de Possível Esteganografia
  • Arquivos com tamanho incompatível com o conteúdo aparente
  • Histogramas de cores com padrões anormais
  • Clusters de disco marcados como "livres" mas contendo dados
  • Pacotes de rede com campos de protocolo preenchidos de forma não padrão
  • Metadata de arquivos com informações suspeitas

18.4 Usando Hashes

Aplicações práticas de hash para verificação de integridade e proteção de senhas.

18.4.1 Arquivos de Hash e Mídia Digital

A integridade assegura que os dados e informações estejam completos e inalterados no momento da aquisição. Isso é crucial quando um usuário baixa um arquivo da Internet ou quando um avaliador forense busca evidências em mídia digital.

Cisco IOS Image Verification

Para verificar a integridade das imagens IOS, a Cisco oferece checksums MD5 e SHA no site de download de software. 

# Verificar checksum da imagem IOS
Router# verify /md5 flash:image.bin
MD5 of flash:image.bin: 4ehiDx67NMop9

# Comparar com o valor disponível no site da Cisco
Site Cisco MD5: 4ehiDx67NMop9 ✅
# Se coincidirem, a imagem não foi violada!
Processo de Verificação
  1. Baixar imagem IOS do site Cisco
  2. Obter o checksum MD5/SHA fornecido
  3. Calcular o hash do arquivo baixado
  4. Comparar com o valor oficial
  5. Se coincidirem → arquivo íntegro ✅
Importância: A verificação de hash garante que ninguém violou ou modificou o arquivo de imagem do IOS durante o download ou armazenamento.

18.4.3 Senhas de Hash

Os algoritmos hash transformam qualquer quantidade de dados em um hash digital de tamanho fixo. Ninguém pode reverter um hash digital para descobrir a entrada original. Se a entrada mudar completamente, o resultado será um hash diferente.

Isso funciona perfeitamente para proteger senhas. Um sistema precisa armazenar uma senha de modo a protegê-la, mas que ainda possa verificar se a senha do usuário está correta.

Cadastro de Conta

Usuário cria conta
com senha: MinhaSenha123

hash = SHA-256("MinhaSenha123")
Armazenado: 8b0bDx67NMop9...

✅ O sistema NUNCA armazena a senha original!

Autenticação

Usuário digita senha:
MinhaSenha123

hash_calculado = SHA-256("MinhaSenha123")
Comparar com hash armazenado

Se coincidirem → ACESSO CONCEDIDO ✅

Vantagem: A senha é conhecida apenas pelo usuário que a definiu. Mesmo se o banco de dados for comprometido, os hashes não revelam as senhas originais.

18.4.4 Como Decifrar Hashes

Para decifrar um hash, um invasor deve adivinhar a senha. Os dois principais ataques usados para adivinhar senhas são:

Ataque de Dicionário

Usa um arquivo que contém palavras, frases e senhas comuns. O arquivo tem os hashes pré-calculados. 

# Dicionário comum
admin → 5f4dcc3b5aa765d61d8327deb882cf99
123456 → e10adc3949ba59abbe56e057f20f883e
password → 5e884898da28047151d0e56f8dc6292773603d0d6aabbdd62a11ef721d1542d8

Se o hash corresponder a qualquer entrada do dicionário → senha descoberta!

Ataque de Força Bruta

Tenta todas as combinações possíveis de caracteres até determinado tamanho. 

# Tentativas sequenciais
a, b, c, ..., z,
aa, ab, ac, ..., zz,
aaa, aab, ..., etc.
Limitação: Consome muito tempo e energia do processador. Em teoria, é apenas uma questão de tempo até descobrir a senha.
Medidas de Proteção
  • ✅ Usar senhas longas (> 12 caracteres)
  • ✅ Combinar letras maiúsculas, minúsculas, números e símbolos
  • ✅ Evitar palavras comuns do dicionário
  • ✅ Usar salt (valor aleatório adicionado antes do hash)

18.4.7 Salting

O salting torna o hash de senhas mais seguro. Um SALT é uma cadeia de caracteres aleatória adicionada à senha antes do hash.

Problema sem salt: Se dois usuários têm a mesma senha, eles terão os mesmos hashes, facilitando a identificação de senhas comuns.
Usuário 1: "senha123" 
Salt: QxLUF1bIAdEQX
String: "senha123" + "QxLUF1bIAdEQX"
Hash: b3bad1e532f4057753a4b8d7cef293e4
Usuário 2: "senha123" 
Salt: R9PelC7sxQXb8
String: "senha123" + "R9PelC7sxQXb8"
Hash: 713c7beb54841a26a7c81eb06d6cf066

✅ Resultado: A mesma senha "senha123" produz hashes completamente diferentes devido ao salt exclusivo!

O salt não precisa ser confidencial, pois é um valor aleatório armazenado junto com o hash.

18.4.8 Implementação de Salting

Para gerar o salt, utiliza-se um CSPRNG (Cryptographically Secure Pseudo-Random Number Generator) - Gerador Numérico Pseudo Randômico de Criptografia Segura.

Características do CSPRNG: Gera números aleatórios com alto nível de aleatoriedade, completamente imprevisíveis e criptograficamente confiáveis.
Para Armazenar uma Senha
  1. Use CSPRNG para gerar um salt longo e aleatório
  2. Adicione o salt no início da senha
  3. Execute o hash com função criptográfica padrão (ex: SHA-256)
  4. Salve o salt e o hash como parte do registro do banco de dados do usuário
Registro: [salt:QxLUF1bIAdEQX][hash:b3bad1e532f4...]
Para Validar uma Senha
  1. Recupere o salt e hash do usuário no banco de dados
  2. Adicione o salt à senha fornecida e calcule o hash com a mesma função
  3. Compare o hash calculado com o hash armazenado
  4. Se não corresponderem → senha incorreta
if hash(senha_input + salt) == hash_armazenado:
✅ Acesso concedido
Recomendações para Implementação Bem-Sucedida
Salt deve ser exclusivo para cada senha de usuário
Nunca reutilize um salt
Tamanho do salt deve corresponder ao tamanho da saída da função hash
Sempre execute o hash no servidor em aplicações web
Key Stretching (Alongamento de Chave):

Técnica que torna as tentativas de descobrir senhas muito mais lentas, aplicando a função hash milhares de vezes. Isso impede que hardware de ponta (que pode calcular bilhões de hashes por segundo) seja eficaz em ataques de força bruta.

Exemplo de Key Stretching (PBKDF2) 
# PBKDF2 - Password-Based Key Derivation Function 2
hash = pbkdf2(senha, salt, iteracoes=100000, comprimento=256)

# O hash é calculado 100.000 vezes, tornando o processo mais lento
# Um invasor precisaria de 100.000x mais tempo para cada tentativa!

18.4.9 Como Evitar Ataques

O salting impede que um invasor use ataques de dicionário para adivinhar senhas. O salting também torna impossível o uso de tabelas de pesquisa e rainbow tables para decifrar hashes.

Tabelas de Pesquisa

Armazenam hashes de senhas pré-calculados em um dicionário de senha, juntamente com a senha correspondente.

📊 Processa centenas de pesquisas de hash por segundo. Extremamente eficiente, mas vulnerável ao salting.

Permite atacar vários hashes simultaneamente sem tabela pré-calculada.

Sacrificam velocidade de quebra para diminuir o tamanho das tabelas.

18.5.1 Uso de Assinaturas Digitais

As assinaturas digitais são uma técnica matemática usada para fornecer autenticidade, integridade e não repúdio. As assinaturas digitais usam criptografia assimétrica.

Autenticidade

A assinatura não pode ser falsificada

🔐 Fornece prova de que o signatário, e ninguém mais, assinou o documento.

Inalterável

Após assinado, não pode ser alterado

📝 Qualquer modificação no documento invalida a assinatura digital.

Não Reutilizável

Assinatura é única por documento

📄 A assinatura do documento não pode ser transferida para outro documento.

Não Repúdio

Signatário não pode negar autoria

⚖️ Serve como prova legal de que o intercâmbio de dados ocorreu.

Algoritmos de Assinatura Digital (DSS)
DSA

Digital Signature Algorithm

Padrão original para gerar pares de chaves públicas/privadas e para gerar/verificar assinaturas digitais.

RSA

Rivest-Shamir-Adleman

Algoritmo assimétrico comumente usado para gerar e verificar assinaturas digitais. Amplamente adotado.

ECDSA

Elliptic Curve Digital Signature Algorithm

Variante mais recente do DSA com eficiência computacional, tamanhos de assinatura pequenos e largura de banda mínima.

PKCS (Public Key Cryptography Standards):

Na década de 1990, a RSA Security Inc. publicou 15 padrões PKCS. Embora não sejam padrões oficiais do setor, são amplamente reconhecidos e relevantes para organizações como IETF e PKIX.

O governo dos EUA especifica que software disponível para download na internet deve ser assinado e verificado digitalmente para garantir que não foi adulterado e que é originado de fonte confiável.

18.6.2 A Infraestrutura de Chave Pública

A Infraestrutura de Chave Pública (PKI) consiste em especificações, sistemas e ferramentas usados para criar, gerenciar, distribuir, usar, armazenar e revogar certificados digitais.

Autoridade de Certificação (CA): Organização que cria certificados digitais vinculando uma chave pública a uma identificação confirmada, como um site ou indivíduo.
Principais Elementos da PKI
1. Certificado PKI

Contém a chave pública de uma entidade ou indivíduo, sua finalidade, a CA que validou e emitiu, o intervalo de validade e o algoritmo usado para criar a assinatura.

2. Armazenamento de Certificados

Reside em um computador local e armazena certificados emitidos e chaves privadas.

3. Autoridade de Certificação (CA)

Terceiro confiável que emite certificados PKI após verificar identidade. Assina os certificados usando sua chave privada.

4. Banco de Dados de Certificados

Armazena todos os certificados aprovados pela autoridade de certificação.

Interoperabilidade dos Elementos PKI

1. Emissão

Bob solicita certificado da CA → CA autentica Bob → Armazena no banco de dados

2. Troca

Bob se comunica com Alice usando seu certificado PKI

3. Verificação

Alice consulta CA confiável para validar o certificado de Bob

Nota: Uma Autoridade de Registro (RA) é uma autoridade subordinada certificada por uma CA raiz para emitir certificados para usos específicos.

Exemplos de Autoridades de Certificação: IDentrust, DigiCert, Sectigo, GlobalSign, GoDaddy. Let's Encrypt é uma CA sem fins lucrativos que oferece certificados gratuitamente.

18.7.2 Transações de rede criptográfica

Um analista de segurança deve ser capaz de reconhecer e resolver possíveis problemas relacionados à permissão de soluções relacionadas à PKI na rede corporativa.

Risco: O aumento do tráfego SSL/TLS representa um grande risco de segurança porque o tráfego é criptografado e não pode ser interceptado e monitorado por meios normais.

Usuários podem introduzir malware ou vazar informações confidenciais através de uma conexão SSL/TLS. Atores de ameaças podem usar SSL/TLS para introduzir:

  • Violações de conformidade regulatória
  • Vírus e malware
  • Perda de dados
  • Tentativas de intrusão na rede
Problemas relacionados à validação de certificados

Quando ocorrem problemas de validação, os navegadores exibem avisos de segurança. Os principais problemas incluem:

Intervalo de datas de validade

Certificados X.509v3 especificam datas "não antes" e "não depois". Se a data atual estiver fora do intervalo, o navegador exibe uma mensagem de erro.

Certificados expirados podem ser resultado de supervisão do administrador ou refletir condições mais graves.

Erro de validação de assinatura

Se o navegador não conseguir validar a assinatura do certificado, não há garantia de que a chave pública seja autenticada.

A validação falha se o certificado raiz da hierarquia da CA não estiver disponível no armazenamento de certificados do navegador.

18.7.3 Criptografia e monitoramento de segurança

O monitoramento de rede torna-se mais desafiador quando os pacotes são criptografados. Os analistas de segurança devem estar cientes desses desafios e enfrentá-los da melhor forma possível.

Desafio: Quando VPNs site a site são usadas, o IPS deve ser posicionado para monitorar o tráfego não criptografado. O aumento do uso de HTTPS introduz tráfego HTTP criptografado de ponta a ponta via TLS/SSL.
Ações que um analista de segurança pode tomar:
Configure regras para distinguir entre tráfego SSL e não-SSL, HTTPS e não-HTTPS
Melhore a segurança através da validação de certificados utilizando CRLs e OCSP
Implemente proteção antimalware e filtragem de URL de conteúdo HTTPS
Implante um Cisco SSL Appliance para descriptografar tráfego SSL e enviar para dispositivos IPS
Como a criptografia afeta as investigações de segurança
1. Ataques direcionados a algoritmos

Após o algoritmo ser quebrado e o invasor obter as chaves, todos os dados criptografados capturados podem ser descriptografados, expondo dados privados.

2. Dados escondidos à vista

Tráfego de comando e controle criptografado com TLS/SSL não pode ser visto por um firewall. O invasor pode continuar usando comandos criptografados para infectar mais computadores e criar botnets.

Detecção: Tráfego TLS/SSL anômalo pode ser detectado descriptografando o tráfego e comparando com assinaturas de ataque conhecidas. Este é um processo difícil e demorado.

18.8 Resumo de criptografia

O que aprendi neste módulo?

Existem duas classes de criptografia usadas para fornecer confidencialidade de dados: assimétrica e simétrica.

  • Algoritmos simétricos (DES, 3DES, AES) - cada parte conhece a chave pré-compartilhada. Comumente usados com tráfego VPN por usarem menos recursos de CPU. Classificados como cifras de bloco ou fluxo.
  • Algoritmos assimétricos (RSA, PKI) - usam chave pública e privada. Exemplos de protocolos: IKE, SSL, SSH, PGP. Algoritmos: DSS, DSA, RSA, ElGamal, curvas elípticas.
  • Confidencialidade assimétrica: Chave pública (Criptografar) + Chave Privada (Descriptografar) = Confidencialidade
  • Autenticação assimétrica: Chave Privada (Criptografar) + Chave Pública (Descriptografar) = Autenticação
  • Diffie-Hellman (DH): Algoritmo que permite gerar uma chave secreta compartilhada idêntica sem comunicação prévia. Usado em VPN IPsec e SSH.

Mascaramento de dados: Substitui informações confidenciais por versão pública. Métodos: substituição, embaralhamento, anulação.

Esteganografia: Esconde dados (mensagem) em outro arquivo (gráfico, áudio, texto). Vantagem: a mensagem secreta não atrai atenção especial.

Quatro elementos das comunicações seguras: integridade de dados, autenticação de origem, confidencialidade de dados, não repúdio de dados.

Hash: Função matemática unilateral que produz representação condensada de comprimento fixo. Funções conhecidas: MD5 (128-bit), SHA-1, SHA-2, SHA-3.

HMAC: Adiciona autenticação à garantia de integridade usando uma chave secreta adicional como entrada à função hash.

Propriedades da função hash: entrada de qualquer comprimento, saída de tamanho fixo, unidirecional (não reversível), quase sem colisões.

Salt: Entrada adicional adicionada à senha antes do hash, criando resultados diferentes mesmo para senhas idênticas. CSPRNG é a melhor opção para gerar o salt.

Ataques: Dicionário (usa palavras comuns) e força bruta (tenta todas combinações). O salting impede ataques de dicionário e rainbow tables.

Assinaturas digitais: Técnica matemática que fornece autenticidade, integridade e não repúdio. Propriedades: autênticas, inalteráveis, não reutilizáveis, não repudiadas.

Algoritmos DSS: DSA (Digital Signature Algorithm), RSA (Rivest-Shamir-Adleman), ECDSA (Elliptic Curve Digital Signature Algorithm).

Assinatura de código: Garante que o código é autêntico, não foi modificado e a editora publicou inegavelmente.

Certificado digital: Equivalente a um passaporte eletrônico. Usado para autenticar e verificar identidade.

PKI: Infraestrutura de Chave Pública - especificações, sistemas e ferramentas para criar, gerenciar, distribuir, usar, armazenar e revogar certificados digitais.

Classes de certificados: Quanto maior o número da classe (0 a 5), mais confiável é o certificado. Determinado pelo rigor do procedimento de verificação de identidade.

Topologias PKI: Raiz única, CA com certificação cruzada, CA hierárquica.

X.509 v3: Padrão que define o formato do certificado digital. RFC 2527 publicado pelo IETF.

Usos comuns da PKI: Autenticação SSL/TLS, VPNs IPsec, HTTPS, 802.1x, e-mail seguro (S/MIME), mensagens instantâneas, assinatura de código, EFS, cartões inteligentes, proteção de dispositivos USB.

Desafios de monitoramento: Tráfego SSL/TLS criptografado dificulta interceptação. Atores de ameaças podem usar para violações, malware, perda de dados.

Problemas de validação: Intervalo de datas (não antes/não depois) e erros de validação de assinatura.

Conjunto de cifras: Componentes: MAC (HMAC), algoritmo de criptografia, algoritmo de troca de chaves, algoritmo de autenticação.

📖 Resumo Didático: Entendendo a Criptografia de Forma Simples

O que é Criptografia?

Imagine que você quer enviar uma carta secreta para um amigo, mas não quer que ninguém a leia no caminho.

A criptografia é exatamente isso: uma forma de "embaralhar" uma mensagem para que apenas quem tem a "chave" correta consiga entendê-la.

Analogia simples: É como escrever uma mensagem com tinta invisível. Quem recebe precisa saber como revelar a mensagem (passar calor, luz UV, etc.) para ler o que está escrito.

Criptografia Simétrica

Uma chave para trancar e destrancar

Como funciona: Você e seu amigo combinam uma senha secreta antes de qualquer comunicação. Usam a MESMA chave para trancar e destrancar a mensagem.

📦 Exemplo: Cofre com cadeado

Você tranca a mensagem em um cofre com um cadeado. Seu amigo tem uma cópia IDÊNTICA da chave para abrir.

✅ Vantagem: Rápida, usa poucos recursos do computador
⚠️ Desvantagem: Como compartilhar a chave de forma segura?

Criptografia Assimétrica

Duas chaves diferentes: pública e privada

Como funciona: Você tem duas chaves. A pública pode ser entregue a qualquer um (serve para trancar). A privada fica só com você (serve para destrancar).

📦 Exemplo: Caixa de correio

Qualquer um pode colocar uma carta na sua caixa (chave pública), mas só você tem a chave para abrir e ler (chave privada).

✅ Vantagem: Não precisa compartilhar chave secreta antes
⚠️ Desvantagem: Mais lenta, usa mais recursos do computador

As Três Missões da Criptografia

Confidencialidade

"Só você e eu sabemos"

A mensagem só pode ser lida pelo destinatário correto. Se alguém interceptar, verá apenas algo incompreensível.

🔒 Mensagem original: "Te vejo às 18h"
🔐 Criptografada: "8x%#Kd9@2!m"
🔓 Só Bob consegue ler

Integridade

"Ninguém mexeu no caminho"

Garante que a mensagem não foi alterada durante o envio. Se alguém modificar um único caractere, você saberá.

📝 Hash original: "a7f3c8"
📝 Hash recebido: "a7f3c8"
✅ Mensagem íntegra!

Autenticidade

"Fui eu mesmo que enviei"

Prova que a mensagem realmente veio de quem diz ter enviado. Ninguém pode fingir ser você.

✍️ Assinatura digital
🔐 Criptografada com chave privada de Alice
✅ Só Alice poderia ter enviado

Analogias que Ajudam a Entender

Diffie-Hellman = Misturando Cores

Alice e Bob concordam com uma cor pública (amarelo). Cada um escolhe uma cor secreta (Alice=vermelho, Bob=azul). Misturam e trocam. Ao misturar novamente com sua cor secreta, ambos chegam à mesma cor marrom - sem nunca ter trocado a cor marrom diretamente!

+ =

Este é o segredo compartilhado!

Certificado Digital = Passaporte

Um passaporte é emitido por um governo (Autoridade Certificadora). Ele contém sua foto, nome, data de validade. Qualquer pessoa que confia no governo aceita seu passaporte como prova de identidade.

🇧🇷 Emitido por: Governo Brasileiro
👤 Nome: João Silva
📅 Válido até: 2030
🔐 Assinatura digital do governo
Hash = Impressão Digital Única

Cada pessoa tem uma impressão digital única. Mesmo gêmeos idênticos têm impressões diferentes. O hash funciona igual: cada arquivo ou mensagem gera um código único.

Arquivo "foto.jpg" → hash: "a7f3c8d9"
Arquivo "foto.jpg" (alterado 1 pixel) → hash: "9xYp76NM"

Qualquer mudança, por menor que seja, altera completamente o hash!

Salt = Tempero Secreto

Duas pessoas podem usar a mesma senha "123456". Sem salt, ambas teriam o mesmo hash. Com salt (um tempero aleatório), cada uma terá um hash completamente diferente!

Senha "123456" + salt "QxL" → hash "b3bad1e5"
Senha "123456" + salt "R9P" → hash "713c7beb"

Glossário Técnico - Termos Essenciais

AES (Advanced Encryption Standard)

Padrão de criptografia simétrica mais usado hoje. Chaves de 128, 192 ou 256 bits. Usado em Wi-Fi, VPNs, arquivos criptografados.

CA (Certificate Authority)

Autoridade Certificadora. Empresa confiável que emite certificados digitais, verificando identidades (ex: DigiCert, Let's Encrypt).

Cipher Suite

Conjunto de algoritmos usados em uma conexão SSL/TLS. Define como a criptografia, troca de chaves e autenticação serão feitas.

CRL (Certificate Revocation List)

Lista de certificados revogados antes do vencimento. Usada para verificar se um certificado ainda é válido.

Diffie-Hellman (DH)

Algoritmo que permite duas partes criarem uma chave secreta compartilhada sem nunca trocá-la diretamente. Base para VPNs e SSH.

HMAC (Hash-based Message Authentication Code)

Hash com chave secreta. Garante integridade E autenticidade da mensagem. Usado em SSL/TLS, IPsec, SSH.

MD5

Algoritmo de hash de 128 bits. Hoje considerado inseguro. Deve ser evitado. Use SHA-2 ou SHA-3.

OCSP (Online Certificate Status Protocol)

Protocolo que verifica em tempo real se um certificado foi revogado. Mais rápido que CRL.

PKI (Public Key Infrastructure)

Infraestrutura de Chave Pública. Conjunto de políticas, hardware e software para gerenciar certificados digitais.

RSA

Algoritmo assimétrico mais famoso. Baseado na dificuldade de fatorar números grandes. Usado em HTTPS, assinaturas digitais.

Salt

Valor aleatório adicionado à senha antes do hash. Impede que senhas iguais gerem hashes iguais, derrotando ataques de dicionário.

SHA (Secure Hash Algorithm)

Família de algoritmos de hash seguros. SHA-256 produz hash de 256 bits. SHA-3 é a versão mais recente.

X.509

Padrão que define o formato de certificados digitais. Usado em SSL/TLS, S/MIME, IPsec, etc.

Non-repúdio

Propriedade que impede o remetente de negar que enviou uma mensagem. É a "prova legal" da comunicação.
