Módulo 25: Avaliação de Vulnerabilidade de Endpoint

Explicar como as vulnerabilidades do endpoint são avaliadas e gerenciadas.

25.0 Introdução

25.0.1 Por que devo usar este módulo?

Quanto dinheiro uma organização deve gastar em segurança de rede e operações cibernéticas? Como uma organização sabe quanto esforço e recursos deve colocar para manter a rede e os dados seguros? Essas perguntas podem ser respondidas através da avaliação de risco e vulnerabilidade.

Analistas de segurança cibernética e especialistas em segurança usam uma variedade de ferramentas para realizar avaliações de vulnerabilidade. A criação de perfis de rede e dispositivo fornece uma linha de base que serve como ponto de referência para identificar desvios em relação às operações normais.

Da mesma forma, a criação de perfil do servidor é usada para estabelecer o estado operacional aceito dos servidores. As organizações usam o Common Vulnerability Scoring System (CVSS) para ponderar os riscos de uma vulnerabilidade usando uma variedade de métricas.

Nota: Este módulo aborda detalhes da criação de perfis de rede e servidor, CVSS, técnicas de gerenciamento de risco e ISMS (Sistema de Gerenciamento de Segurança da Informação).

25.1 Perfil de Rede e Servidor

25.1.1 Perfilamento de rede

Para detectar incidentes de segurança graves, é importante compreender, caracterizar e analisar informações sobre o funcionamento normal da rede. Todas as redes, servidores e hosts exibem um comportamento típico para um determinado ponto no tempo. A criação de perfis de rede e dispositivo pode fornecer uma linha de base estatística que serve como ponto de referência. Desvios inexplicáveis em relação à linha de base podem indicar um compromisso.

Deve-se ter cuidado ao capturar dados de linha de base para que todas as operações normais de rede sejam incluídas na linha de base. Além disso, é importante que a linha de base seja atual. Não deve incluir dados de desempenho da rede que não façam mais parte do funcionamento normal.

Dica: Ferramentas como NetFlow e Wireshark podem ser usadas para caracterizar características normais de tráfego de rede.

Elementos do Perfil de Rede

Elemento de perfil de rede Descrição
Duração de sessão Este é o tempo entre o estabelecimento de um fluxo de dados e seu término
Taxa de transferência total Quantidade de dados que passam de uma fonte para um destino em um determinado período
Portas usadas Lista de processos TCP ou UDP que estão disponíveis para aceitar dados
Espaço de endereço de ativo crítico Endereços IP ou localização lógica de sistemas ou dados essenciais

25.1.2 Criação de perfil do servidor

A definição de perfil do servidor é usada para estabelecer o estado operacional aceito dos servidores. Um perfil de servidor é uma linha de base de segurança para um determinado servidor. Estabelece os parâmetros de rede, usuário e aplicativo que são aceitos para um servidor específico.

Para estabelecer um perfil de servidor, é importante entender a função que um servidor se destina a executar em uma rede. A partir daí, vários parâmetros operacionais e de uso podem ser definidos e documentados.

Elemento de perfil do servidor Descrição
Portas de escuta Daemons e portas TCP/UDP que normalmente podem ser abertos no servidor
Usuários e contas logados Parâmetros que definem o acesso e o comportamento do usuário
Contas de serviço Definições do tipo de serviço que um aplicativo pode executar
Ambiente de Software Tarefas, processos e aplicativos que têm permissão para serem executados no servidor

25.1.3 Detecção de anomalias de rede

O comportamento da rede é descrito por uma grande quantidade de dados diversos, como os recursos do fluxo de pacotes, os recursos dos próprios pacotes e a telemetria de várias fontes. Uma abordagem para a detecção de ataques de rede é a análise desses dados diversos e não estruturados usando técnicas de análise de Big Data. Isso é conhecido como análise de comportamento de rede (NBA). 

Algoritmo de Detecção de Anomalias - Roteadores de Fronteira

A cada X min:
    Contagem de fluxos com Amostragem 1/Y durante Z seg
    
    Se # de fluxos > N:
        Gerar ALARME!
    Senão:
        Continuar monitoramento normal

Por exemplo, o analista de segurança cibernética poderia fornecer os seguintes valores:

X = 5 minutos Y = 100 Z = 30 segundos N = 500 fluxos

O algoritmo pode ser interpretado como: a cada 5 minutos, obter uma amostragem de 1/100 dos fluxos durante o segundo 30. Se o número de fluxos for maior que 500, gere um alarme.

NBA (Network Behavior Analysis): Utiliza técnicas sofisticadas de aprendizagem estatística e de máquina para comparar as linhas de base de desempenho normais com o desempenho da rede num determinado momento.

A maioria das organizações conecta-se às redes públicas devido à necessidade de acessar a internet. Testes periódicos de segurança são essenciais devido ao grande número de potenciais vulnerabilidades.

25.2 Sistema de Pontuação de Vulnerabilidade Comum (CVSS)

25.2.1 Visão geral do CVSS

O Common Vulnerability Scoring System (CVSS) é uma ferramenta de avaliação de risco projetada para transmitir os atributos comuns e a gravidade das vulnerabilidades em sistemas de hardware e software de computador. A terceira revisão, CVSS 3.0, é uma estrutura aberta e neutra do fornecedor, padrão do setor, para ponderar os riscos de uma vulnerabilidade usando uma variedade de métricas.

Esses pesos combinam-se para fornecer uma pontuação do risco inerente a uma vulnerabilidade. A pontuação numérica pode ser usada para determinar a urgência da vulnerabilidade e a prioridade de abordá-la.

Pontuações Padronizadas

Significativas em todas as organizações

Estrutura Aberta

Significado de cada métrica disponível para todos

Priorização de Risco

Significativa para organizações individuais

FIRST: O Fórum de Equipes de Resposta a Incidentes e Segurança foi designado como guardião do CVSS. A versão 3.1 foi lançada em junho de 2019 com contribuições da Cisco.

Impacto de Disponibilidade (A)

Impacto na acessibilidade de informações e recursos de rede

25.2.4 O Processo CVSS

O CVSS Base Metrics Group foi projetado para avaliar vulnerabilidades de segurança encontradas em sistemas de software e hardware. O processo CVSS usa uma ferramenta chamada Calculadora CVSS v3.1. 

String Vetorial CVSS: CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:L/A:L
Nome da Métrica Iniciais Valores Possíveis
Vetor de Ataque AV N (Rede), A (Adjacente), L (Local), P (Físico)
Complexidade do ataque AC L (Baixo), H (Alto)
Privilégios obrigatórios PR N (Nada), L (Baixo), H (Alto)
Interação de Usuário UI N (Nada), R (Requerido)
Escopo S U (Não alterado), C (Alterado)
Impacto de Confidencialidade C H (Alto), L (Baixo), N (Nada)
Impacto de Integridade I H (Alto), L (Baixo), N (Nada)
Impacto de Disponibilidade A H (Alto), L (Baixo), N (Nada)

25.2.6 Outras fontes de informações sobre vulnerabilidades

CVE - Vulnerabilidades e Exposições Comuns

Dicionário de nomes comuns, na forma de identificadores CVE, para vulnerabilidades conhecidas de segurança cibernética.

  • Fornece uma maneira padrão de pesquisar vulnerabilidades
  • Identificadores CVE permitem acessar correções
  • Usado por serviços de inteligência contra ameaças
  • Aparece em vários logs do sistema de segurança

NVD - National Vulnerability Database

Banco de dados mantido pelo NIST (National Institute of Standards and Technology) do governo dos EUA.

  • Utiliza identificadores CVE
  • Fornece pontuações de ameaças CVSS
  • Detalhes técnicos e entidades afetadas
  • Recursos para investigação adicional

25.3 Gerenciamento Seguro de Dispositivos

25.3.1 Gerenciamento de risco

O gerenciamento de riscos envolve a seleção e especificação de controles de segurança para uma organização. Faz parte de um programa contínuo de segurança da informação em toda a organização que envolve a gestão do risco para a organização ou para indivíduos associados à operação de um sistema.

Processo Contínuo: O gerenciamento de riscos é um processo contínuo, multipasso e cíclico.
Identificação de Risco

Identificar ativos, vulnerabilidades e ameaças

Avaliação de risco

Pontuar, pesar e priorizar riscos

Planejamento de Resposta

Determinar resposta ao risco, planejar ações

Implementação de Resposta

Implementar resposta ao risco

Monitorar e avaliar

Monitoramento contínuo de riscos

O risco é determinado como a relação entre ameaça, vulnerabilidade e a natureza da organização. As perguntas fundamentais da avaliação de risco são:

  • Quem são os atores que querem nos atacar?
  • Quais vulnerabilidades os atores ameaçadores podem explorar?
  • Como seríamos afetados por ataques?
  • Qual é a probabilidade de que diferentes ataques ocorram?

25.3.2 Respostas ao Risco

Resposta ao Risco Descrição
Prevenção de riscos Parar de realizar as atividades que criam risco. Se o risco supera o benefício, a atividade deve ser descontinuada.
Redução de risco Diminuir o risco tomando medidas para reduzir a vulnerabilidade (ex: aplicar patches assim que vulnerabilidades são identificadas).
Compartilhamento de risco Deslocar parte do risco para outras partes (ex: terceirizar operações de segurança, contratar SECaaS, comprar seguro).
Retenção de risco Aceitar o risco e suas consequências. Aceitável para riscos de baixo impacto potencial e custo relativamente alto de mitigação.

25.3.3 Gestão de Vulnerabilidade - Ciclo de Vida

De acordo com o NIST, o gerenciamento de vulnerabilidades é uma prática de segurança projetada para prevenir proativamente a exploração de vulnerabilidades de TI existentes em uma organização. O resultado esperado é reduzir o tempo e o dinheiro gasto para lidar com vulnerabilidades.

Descobrir

Inventário de ativos, identificar vulnerabilidades, desenvolver linha de base

Priorizar Ativos

Categorizar ativos, atribuir valor comercial

Avaliar

Determinar perfil de risco, medir nível de risco comercial

Reparar

Priorizar por risco, resolver vulnerabilidades

Verificar

Auditorias de acompanhamento, verificar eliminação de ameaças

Benefício: O gerenciamento proativo de vulnerabilidades reduz ou elimina o potencial de exploração, envolvendo consideravelmente menos tempo e esforço do que responder após uma exploração.

25.3.4 Gerenciamento de ativos

O gerenciamento de ativos envolve a implementação de sistemas que controlam a localização e a configuração de dispositivos e software em rede em uma empresa. As organizações devem saber quais equipamentos acessam a rede, onde esse equipamento está e que software/dados esses sistemas armazenam.

Descobrir

Inventário automatizado de dispositivos

Priorizar

Categorizar por valor comercial

Avaliar

Medir risco comercial

Reparar/Verificar

Corrigir discrepâncias, auditar

Processo de Gerenciamento de Ativos (NIST):

1. Coletar o estado atual de todos os dispositivos autorizados
2. Especificar o estado desejado dos dispositivos
3. Localizar discrepâncias no estado do dispositivo
4. Corrigir discrepâncias, aceitar riscos ou atualizar políticas

25.3.5 Gerenciamento de dispositivos móveis (MDM)

O gerenciamento de dispositivos móveis, especialmente na era do BYOD (Bring Your Own Device), apresenta desafios especiais para a gestão de ativos. Dispositivos móveis podem ser perdidos, roubados ou adulterados, colocando em risco o acesso a dados e à rede.

Desafios do MDM
  • Dispositivos fora do controle físico da organização
  • Diversidade de dispositivos (iOS, Android, etc.)
  • Dispositivos inerentemente menos seguros
  • Risco de perda ou roubo
Medidas de Segurança
  • Desativar dispositivo perdido remotamente
  • Criptografar dados no dispositivo
  • Autenticação robusta
  • Assumir que dispositivos não são confiáveis inicialmente
Solução: Sistemas MDM como Cisco Meraki Systems Manager permitem configurar, monitorar e atualizar clientes móveis a partir da nuvem.

25.3.6 Gerenciamento de configurações

O gerenciamento de configuração aborda o inventário e o controle das configurações de hardware e software dos sistemas. Configurações seguras de dispositivos reduzem o risco de segurança.

Conforme definido pelo NIST, gerenciamento de configuração:

Compreende uma coleção de atividades focadas no estabelecimento e manutenção da integridade de produtos e sistemas, através do controle dos processos de inicialização, mudança e monitoramento das configurações.

Imagens de Software

Pacote básico + segurança de endpoint + políticas personalizadas

Configurações de Hardware

Interfaces de rede permitidas, tipos de armazenamento externo

Ferramentas de Automação

Puppet, Chef, Ansible, SaltStack

Referência: NIST Special Publication 800-128 - Gerenciamento de Configuração para Segurança de Rede.

25.3.8 Técnicas de gerenciamento de patch

Baseado em agente

Requer que um agente de software esteja sendo executado em cada host a ser corrigido.

  • Agente informa se software vulnerável está instalado
  • Comunica-se com o servidor de gerenciamento de patches
  • Determina e instala patches necessários
  • Executa com privilégios suficientes
  • ✓ Meio preferido para dispositivos móveis

Varredura sem agente

Servidores de gerenciamento de patches verificam a rede em busca de dispositivos que exigem patches.

  • Servidor determina quais patches são necessários
  • Instala patches nos clientes remotamente
  • Apenas dispositivos em segmentos de rede digitalizados
  • ⚠ Pode ser problema para dispositivos móveis

Monitoramento de rede passivo

Dispositivos que requerem patches são identificados através do monitoramento do tráfego na rede.

  • Servidor monitora e rastreia versão do software
  • Patches implantados conforme necessário
  • Eficaz para software que inclui informações de versão no tráfego
  • ✓ Monitoramento contínuo

25.4 Resumo da Avaliação de Vulnerabilidade de Endpoint

25.4.1 O que aprendi neste módulo?

É importante executar o perfil de rede e dispositivo para fornecer informações estatísticas de linha de base que podem servir como um ponto de referência para o desempenho normal da rede e do dispositivo.

  • Elementos importantes do perfil de rede: duração da sessão, taxa de transferência total, portas usadas e espaço de endereço de ativos críticos
  • Perfil do servidor: estabelece o estado operacional aceito dos servidores (parâmetros de rede, usuário e aplicativo)
  • Detecção de anomalias: análise de Big Data para detecção estatística, comportamental e baseada em regras
  • Avaliação de segurança: Análise de Risco, Avaliação de Vulnerabilidades (OpenVAS, Nessus, Nmap), Teste de Penetração (Metasploit)

O CVSS é uma estrutura aberta de padrão do setor para classificar os riscos de vulnerabilidades usando métricas para calcular uma pontuação composta.

  • Três grupos métricos: Base, Temporal e Ambiental
  • Métricas de explorabilidade: Vetor de Ataque (AV), Complexidade (AC), Privilégios (PR), Interação do Usuário (UI), Escopo (S)
  • Classificações: Nenhum (0), Baixo (0.1-3.9), Médio (4.0-6.9), Alto (7.0-8.9), Crítico (9.0-10.0)
  • Regra geral: Vulnerabilidades acima de 3.9 devem ser resolvidas
  • Fontes adicionais: CVE (Common Vulnerabilities and Exposures) e NVD (National Vulnerability Database)

O gerenciamento de riscos envolve a seleção e especificação de controles de segurança para uma organização.

  • Quatro respostas ao risco: Prevenção (interromper atividade), Redução (mitigar), Compartilhamento (terceirizar/seguro), Retenção (aceitar)
  • Ciclo de vida de vulnerabilidade: Descobrir → Priorizar ativos → Avaliar → Relatar → Corrigir → Verificar
  • Gerenciamento de ativos: Controla localização e configuração de dispositivos e software
  • MDM: Configurar, monitorar e atualizar clientes móveis a partir da nuvem (ex: Cisco Meraki)
  • Gerenciamento de configuração: Inventário e controle de configurações de hardware/software (Puppet, Chef, Ansible)
  • Gerenciamento de patches: Aquisição, distribuição, instalação e verificação de patches
  • Técnicas de patch: Baseado em agente, Varredura sem agente, Monitoramento de rede passivo
Conclusão do Módulo 25: Você aprendeu sobre criação de perfis de rede e servidor, sistema CVSS para pontuação de vulnerabilidades e técnicas de gerenciamento seguro de dispositivos.

Fixação do Conteúdo: Glossário + Casos de Uso

Glossário Técnico

Referência rápida dos principais termos do Módulo 25

Perfil de Rede

Linha de base estatística do comportamento normal da rede que serve como referência para detectar desvios e possíveis comprometimentos.

Duração da Sessão

Tempo entre o estabelecimento de um fluxo de dados e seu término.

Rendimento Total (Throughput)

Quantidade de dados que trafega de uma fonte para um destino em um determinado período.

Portas de Escuta

Processos TCP/UDP disponíveis para aceitar dados em um servidor.

NBA (Network Behavior Analysis)

Análise de comportamento de rede usando técnicas de Big Data e machine learning para detectar anomalias.

Avaliação de Vulnerabilidade

Teste automatizado que verifica servidores e redes em busca de vulnerabilidades como patches ausentes ou portas desnecessárias.

Teste de Penetração (Pentest)

Ataque simulado autorizado que não apenas identifica vulnerabilidades mas as explora para determinar impacto potencial.

CVSS (Common Vulnerability Scoring System)

Framework padrão do setor para classificar gravidade de vulnerabilidades usando métricas padronizadas.

Vetor de Ataque (AV)

Métrica CVSS que reflete a proximidade do agente de ameaça ao componente vulnerável (Rede, Adjacente, Local, Físico).

CVE (Common Vulnerabilities and Exposures)

Dicionário público de identificadores padronizados para vulnerabilidades de segurança conhecidas.

NVD (National Vulnerability Database)

Banco de dados do NIST que complementa CVE com pontuações CVSS, detalhes técnicos e recursos de investigação.

Gerenciamento de Patches

Processo de identificar, adquirir, distribuir, instalar e verificar patches de software para mitigar vulnerabilidades.

MDM (Mobile Device Management)

Sistemas que permitem configurar, monitorar e atualizar dispositivos móveis remotamente, incluindo ações como bloqueio remoto.

ISMS (Information Security Management System)

Sistema para identificar, analisar e lidar com riscos de segurança da informação.

Casos de Uso Humanizados

Situações reais que ilustram a aplicação dos conceitos do módulo

🏥 Caso 1: O Hospital que Detectou um Ataque Silencioso

Perfil de Rede & Detecção de Anomalias

A situação: O Hospital Santa Clara mantinha um perfil de rede há 6 meses. Tudo funcionava normalmente até que, em um domingo à noite, o sistema de criação de perfis detectou algo estranho.

O que aconteceu: Um funcionário clicou em um link malicioso. O malware começou a se espalhar lateralmente entre os computadores da administração - algo que nunca acontecia no comportamento normal da rede, já que clientes não se comunicavam diretamente entre si.

A linha de base salvou o dia: O desvio foi detectado em menos de 5 minutos. O time de segurança isolou os hosts infectados antes que o ransomware criptografasse os prontuários dos pacientes.

Ligação com a teoria: O aumento de tráfego host-to-host é um indicador clássico de malware se espalhando lateralmente - exatamente como visto na seção 25.1.1.

Desvio detectado: Tráfego host-to-host ↑ 450%

Tempo de resposta: 5 minutos

🏦 Caso 2: O Banco que Priorizou a Vulnerabilidade Certa

CVSS & Priorização de Riscos

A situação: O Banco Sul resolveria todas as vulnerabilidades em ordem de descoberta. O time de TI ficava sobrecarregado e vulnerabilidades críticas demoravam semanas para serem corrigidas.

A mudança: Adotaram o CVSS para classificar vulnerabilidades. Um dia receberam dois alertas: uma vulnerabilidade com score 4.2 (Médio) e outra com score 9.4 (Crítico).

O resultado: A vulnerabilidade crítica (que permitia execução remota de código no servidor de internet banking) foi corrigida em 4 horas. A de score médio aguardou a próxima janela de manutenção. Um ataque foi tentado no dia seguinte - mas a porta já estava fechada.

Ligação com a teoria: Vulnerabilidades acima de 3.9 devem ser resolvidas - quanto maior o score, maior a urgência (seção 25.2.5).

Score CVSS 9.4: Crítico

Tempo de correção: 4 horas

📱 Caso 3: A Consultoria que Perdeu um iPad (e quase os dados do cliente)

MDM & Gerenciamento de Patches

A situação: A Consultoria Nexus adotou BYOD (Bring Your Own Device) sem um sistema MDM adequado. Um consultor perdeu seu iPad no aeroporto com dados confidenciais de clientes.

O problema: Sem MDM, não havia como desativar o dispositivo remotamente ou criptografar os dados. Além disso, seu notebook pessoal (também usado para trabalho) tinha uma vulnerabilidade crítica no Java com score CVSS 9.8 - mas o patch não foi aplicado porque o funcionário raramente ligava a VPN.

A solução implementada: Após o incidente, a empresa adotou:

  • Cisco Meraki MDM - para gerenciar dispositivos móveis remotamente
  • Patch Management baseado em agente - garantindo que dispositivos fora da rede também recebessem patches
  • Política de retenção de risco - para dispositivos que não podiam ser totalmente controlados

Ligação com a teoria: MDM permite desativar dispositivo perdido e criptografar dados (25.3.5); abordagem baseada em agente é preferida para dispositivos móveis (25.3.8).

Antes: Sem MDM

Depois: Cisco Meraki + Agente de Patch

🏪 Caso 4: A Loja de Conveniência que Escolheu Não Investir (e estava certa)

Respostas ao Risco (Retenção)

A situação: A rede de lojas "Mini-Mercado Express" tinha 15 lojas físicas. O sistema de caixa ficava em um computador isolado (sem acesso à internet) e processava apenas transações presenciais.

A avaliação de risco mostrou:

  • Probabilidade de ataque cibernético: Muito Baixa (sistema isolado)
  • Custo de mitigação completa (firewall, IDS, monitoramento 24/7): R$ 50.000/mês
  • Impacto potencial de um ataque: R$ 10.000 (perda de um dia de vendas)

A decisão: A empresa escolheu RETAÇÃO DE RISCO - aceitar o risco e não tomar ação adicional, pois o custo de mitigação superava em muito o potencial de perda.

Ligação com a teoria: Retenção de risco é aceitável quando o impacto potencial é baixo e o custo de mitigação é alto (seção 25.3.2).

Decisão: Retenção de Risco

Custo mitigação >> Impacto potencial
