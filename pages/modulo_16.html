Firewalls de Política com Base em Zona

Implemente o firewall de política baseado em zona usando o CLI. Aprenda a evolução dos firewalls clássicos para o modelo ZPF e domine a segurança baseada em zonas.

16.0 Introdução

Firewalls de política baseados em zona (ZPFs) são um passo evolucionário além dos firewalls clássicos. Enquanto a configuração de segurança baseada em Firewalls clássicas em interfaces de roteador, um ZPF permite que as interfaces sejam atribuídas às zonas. As políticas de segurança são definidas com base na zona e nas relações de segurança entre as zonas. Múltiplas interfaces podem ser feitas membros de uma zona e políticas de zona serão aplicadas a essas interfaces.

Objetivo do Módulo: Implementar firewall de política baseado em zona usando CLI.

16.1 Visão Geral do ZPF

16.1.1 Benefícios de um ZPF

As principais motivações para profissionais de segurança de rede migram para o modelo ZPF são estrutura e facilidade de uso. A abordagem estruturada é útil para documentação e comunicação. A facilidade de uso torna as implementações de segurança de rede mais acessíveis. 

✓ Não é dependente de ACL
✓ Postura de segurança: bloquear a menos que explicitamente permitido
✓ Políticas fáceis de ler com Cisco C3PL
✓ Interfaces virtuais/físicas agrupadas em zonas
✓ Políticas aplicadas ao tráfego unidirecional entre zonas

Firewall Clássico vs ZPF

Ambos os modelos podem ser ativados simultaneamente, mas não combinados em uma única interface. Uma interface não pode ser configurada como membro de zona e para inspeção IP ao mesmo tempo.

16.1.2 Projeto ZPF

Projetar ZPFs envolve várias etapas fundamentais para garantir uma implementação segura e eficiente.

Etapa 1
Determine as zonas

O administrador se concentra na separação da rede em zonas. Zonas estabelecem as fronteiras de segurança de uma rede. Por exemplo, a rede pública seria uma zona e a rede interna seria outra zona.

Etapa 2
Estabelecer políticas entre zonas

Para cada par de zonas "fonte-destino", defina as sessões que os clientes nas zonas de origem podem solicitar. As políticas são unidirecionais e definidas com base em pares de zona.

Etapa 3
Projete a infraestrutura física

Leve em conta os requisitos de segurança e disponibilidade. Inclui ditar o número de dispositivos entre zonas mais seguras e menos seguras e determinar dispositivos redundantes.

Etapa 4
Identificar subconjuntos dentro de zonas e mesclar requisitos de tráfego

Para cada dispositivo de firewall, identificar subconjuntos de zonas conectadas às interfaces e mesclar os requisitos de tráfego para essas zonas.

Operação ZPF e C3PL

O Cisco Common Classification Policy Language (C3PL) é um método estruturado para criar políticas de tráfego com base em eventos, condições e ações. Isto fornece escalabilidade porque uma política afeta todo o tráfego dado, em vez de precisar de ACLs múltiplos e ações de inspeção para tipos diferentes de tráfego. 

class-map type inspect match-any INTERNAL_TRAFFIC
 match protocol http

 match protocol dns
!
policy-map type inspect ZONE_POLICY
 class type inspect INTERNAL_TRAFFIC
  inspect

  drop
!
zone-pair security ZP-INTERNET source INSIDE destination OUTSIDE
 service-policy type inspect ZONE_POLICY

16.2 Operação ZPF

16.2.1 Ações ZPF

As políticas identificam ações que o ZPF executará no tráfego de rede. Três ações possíveis podem ser configuradas para processar o tráfego por protocolo, zonas de origem e destino (pares de zonas) e outros critérios.

Inspect

Realiza inspeção de pacotes de estado de Cisco IOS. Rastreia conexões e sessões.

Drop

Análogo a deny em ACL. Opção log registra pacotes rejeitados.

Pass

Análogo a permit em ACL. Não rastreia estado das conexões.

policy-map type inspect ZONE-POLICY
 class type inspect TRAFFIC-CLASS
  inspect   ! Realiza inspeção stateful
 class type inspect DROP-CLASS
  drop log  ! Descarta e registra log
 class class-default
  pass      ! Permite sem inspeção

16.2.3 Regras para Tráfego para a Zona Self (Autozona)

A autozona é o próprio roteador e inclui todos os endereços IP atribuídos às interfaces do roteador. Este é o tráfego que se origina no roteador ou é endereçado a uma interface de roteador (gerenciamento SSH, protocolos de roteamento).

16.3 Configure um ZPF

16.3.1 Visão Geral da Configuração

Existem cinco etapas no processo de configuração de um ZPF. Primeiro as zonas são criadas. Em seguida, um ou mais mapas de classe são criados para especificar o tráfego que deve ser associado a uma política. Depois, as políticas são criadas que associam o tráfego de mapa de classe com as ações de pass, drop ou inspect. É então necessário criar pares de zona que serão associados a mapas de política. Finalmente, as interfaces são associadas a zonas. Neste ponto, a política do ZPF está ativa.

Importante: A sequência de etapas não é obrigatória, mas algumas configurações devem ser concluídas em ordem. Por exemplo, configure um class-map antes de atribuí-lo a um policy-map.

16.3.2 Etapa 1: Crie as zonas

Antes de criar as zonas, responda a algumas perguntas:

  • Quais interfaces devem ser incluídas nas zonas?
  • Qual será o nome para cada zona?
  • Que tráfego é necessário entre as zonas e em que direção?

Crie as zonas com o comando zone security:

Router(config)# zone security zone-name

R1(config)# zone security PRIVATE
R1(config-sec-zone)# exit
R1(config)# zone security PUBLIC
R1(config-sec-zone)# exit

16.3.3 Etapa 2: Identifique o tráfego (Class-Map)

Use um mapa de classe para identificar o tráfego a que uma política será aplicada. Uma classe define um conjunto de pacotes com base em condições de "match".

Parâmetro Descrição
match-any Pacotes devem atender a UM dos critérios
match-all Pacotes devem atender a TODOS os critérios 
Router(config)# class-map type inspect [match-any | match-all] class-map-name
Router(config-cmap)# match access-group {acl-# | acl-name}
Router(config-cmap)# match protocol protocol-name
Router(config-cmap)# match class-map class-map-name

R1(config)# class-map type inspect match-any HTTP-TRAFFIC
R1(config-cmap)# match protocol http
R1(config-cmap)# match protocol https
R1(config-cmap)# match protocol dns

O tráfego pode corresponder a qualquer uma das declarações para se tornar membro da classe HTTP-TRAFFIC.

16.3.4 Etapa 3: Defina uma ação (Policy-Map)

Use um mapa de política para definir que ação deve ser tomada para o tráfego membro de uma classe.

Ação Descrição
inspect Controle de tráfego baseado em estado. Mantém informações de sessão e permite tráfego de retorno.
drop Descarta tráfego não desejado (ação padrão para class-default).
pass Ação sem estado. Permite encaminhamento unidirecional, sem rastrear conexões. 
Router(config)# policy-map type inspect policy-map-name
Router(config-pmap)# class type inspect class-map-name
Router(config-pmap-c)# {inspect | drop | pass}

R1(config)# policy-map type inspect PRIV-TO-PUB-POLICY
R1(config-pmap)# class type inspect HTTP-TRAFFIC
R1(config-pmap-c)# inspect
Drop padrão: Há um drop explícito aplicado pelo IOS ao fim de cada policy-map, listado como class class-default.

16.3.5 Etapa 4: Identifique um Zone-Pair e combine a uma política

Crie um par de zona e associe esse par a um mapa de política.

Parâmetro Descrição
source source-zone-name Zona de origem do tráfego
destination destination-zone-name Zona de destino do tráfego
self Zona definida pelo sistema (próprio roteador) 
Router(config)# zone-pair security zone-pair-name source {source-zone-name | self} destination {destination-name}
Router(config-sec-zone-pair)# service-policy type inspect policy-map-name

R1(config)# zone-pair security PRIV-PUB source PRIVATE destination PUBLIC
R1(config-sec-zone-pair)# service-policy type inspect PRIV-TO-PUB-POLICY

16.3.6 Etapa 5: Atribua zonas às interfaces

Associe uma zona a uma interface usando o comando zone-member security. Isso aplicará imediatamente a política de serviço associada à zona. 

Router(config-if)# zone-member security zone-name

R1(config)# interface GigabitEthernet0/0
R1(config-if)# zone-member security PRIVATE
R1(config)# interface Serial0/0/0
R1(config-if)# zone-member security PUBLIC
Importante: Se nenhum service-policy está configurada para a zona, todo o tráfego de trânsito será descartado.

A política de serviço agora está ativa. O tráfego HTTP, HTTPS e DNS da zona privada para a zona pública será inspecionado. O tráfego da zona pública para a privada só será permitido se for parte de sessões iniciadas por hosts da zona privada.

16.3.10 Considerações de Configuração do ZPF

  • O roteador nunca filtra o tráfego entre interfaces na mesma zona.
  • Uma interface não pode pertencer a várias zonas.
  • ZPF pode coexistir com Firewall clássico, mas não na mesma interface.
  • O tráfego nunca flui entre interface com zona e interface sem zona.
  • A política entre zonas padrão é descartar todo o tráfego a menos que explicitamente permitido.
  • O comando zone-member não protege o próprio roteador, a menos que pares de zona usem a zona self.

16.5 Entendendo ZPF de Forma Simples

A Analogia do Prédio Corporativo

Imagine um prédio corporativo com diferentes áreas:

  • Área administrativa (funcionários internos) → Zona Privada
  • Área de recepção e servidores (visitantes autorizados) → Zona DMZ
  • Rua e estacionamento público (qualquer pessoa) → Zona Pública

O ZPF (Firewall de Política Baseado em Zona) funciona como um sistema de segurança do prédio que decide:

  • Quem pode entrar em cada área
  • O que pode ser feito dentro de cada área
  • Como as pessoas se movem entre as áreas

Assim como um segurança no prédio, o ZPF não confia em ninguém até que uma regra específica permita a passagem.

ZPF = Segurança por áreas
cada área tem suas regras

Glossário: Termos Técnicos Explicados

Uma tradução simples para os conceitos que você vai encontrar no mundo dos firewalls ZPF:

Zona

O que parece ser: Um grupo de computadores que compartilham o mesmo nível de confiança.

Analogia: Como os andares de um prédio. O térreo (público) é diferente do andar dos diretores (privado).

Exemplo: Zona Privada (funcionários), Zona Pública (internet), Zona DMZ (servidores web).

Tráfego de Trânsito

O que parece ser: Dados que passam pelo roteador, vindos de uma zona e indo para outra.

Analogia: Uma pessoa que entra no prédio (vem da rua), passa pela recepção e vai para uma sala específica.

Exemplo: Um funcionário da zona privada acessando um site na internet (zona pública).

Zona Self (Autozona)

O que parece ser: O próprio roteador/firewall.

Analogia: A portaria do prédio onde os seguranças ficam. É o próprio local de controle.

Exemplo: Um administrador fazendo SSH para configurar o roteador (tráfego destinado à zona self).

Inspect (Inspecionar)

O que parece ser: O firewall rastreia a conversa inteira, não apenas pacotes isolados.

Analogia: Um segurança que anota quem entrou e só deixa sair quem ele viu entrar. Ele "lembra" das pessoas.

Exemplo: Você acessa um site HTTPS. O firewall lembra que VOCÊ iniciou a conexão e permite a resposta do site.

Drop (Descartar)

O que parece ser: O firewall joga fora os pacotes, como se nunca tivessem chegado.

Analogia: Um segurança que não deixa ninguém passar sem crachá. Se não tem permissão, é barrado na porta.

Exemplo: Alguém da internet tentando acessar seu computador pessoal. O firewall descarta.

Pass (Permitir)

O que parece ser: O firewall deixa o pacote passar, mas não guarda registro da conversa.

Analogia: Uma porta giratória sem catraca. As pessoas passam, mas ninguém anota quem entrou ou saiu.

Exemplo: Tráfego VPN (IPsec) onde a segurança já é tratada pela própria conexão criptografada.

Zone-Pair (Par de Zona)

O que parece ser: A definição de uma via de mão única entre duas zonas.

Analogia: Uma ponte que conecta dois bairros com uma placa indicando o sentido permitido.

Exemplo: Zone-pair PRIVADA → PÚBLICA (permite saída, mas não entrada).

C3PL (Cisco Common Classification Policy Language)

O que parece ser: Uma linguagem organizada para escrever regras de firewall.

Analogia: Um formulário padronizado que todo segurança preenche para liberar acesso, garantindo que ninguém esqueça informações importantes.

Exemplo: Class-map → Policy-map → Zone-pair. Uma estrutura organizada como um checklist.

Regras do ZPF em Português Claro

Tráfego de Trânsito (atravessa o roteador)

✓ NENHUMA interface em zona → Tráfego passa livre (como rua sem porteiro).

✗ APENAS UMA interface em zona → Tráfego descartado (como alguém com crachá tentando entrar num prédio sem porteiro).

✓ AMBAS na MESMA zona → Tráfego passa (pessoas no mesmo andar podem circular).

✗ Zonas DIFERENTES sem política → Tráfego descartado (como tentar entrar em outro andar sem permissão).

🔍 Zonas DIFERENTES com política → Inspecionado (o segurança verifica e registra).

Zona Self (tráfego para o roteador)

✓ SEMPRE permitido quando o roteador é fonte ou destino.

🔍 Exceção: Se existe política específica, ela é aplicada.

Isso permite que você gerencie o roteador (SSH, SNMP) mesmo com firewall ativo.

Regra de Ouro do ZPF: "Confie em ninguém, verifique tudo, permita apenas o necessário." O ZPF começa bloqueando TUDO e só libera o que você explicitamente autorizar.

Perguntas Frequentes (Em Português Claro)

Resposta simples: É como comparar um guarda com uma lista de nomes (firewall clássico) com um sistema de crachás por áreas (ZPF).

No ZPF, você agrupa interfaces em zonas (ex: "funcionários", "visitantes", "rua") e cria regras entre zonas. Se amanhã você adicionar uma nova interface na zona "funcionários", as regras já se aplicam automaticamente. É mais organizado, escalável e fácil de entender.

Analogia: Imagine um porteiro que anota quem entra no prédio. Quando a pessoa sai, ele confere se ela realmente entrou antes. Um firewall "stateful" faz isso com dados: ele lembra que VOCÊ pediu para ver um site e permite que o site te responda. Sem isso, ele teria que bloquear a resposta porque "alguém de fora" está tentando falar com você.

O ZPF com a ação inspect é stateful. A ação pass não guarda esse registro (stateless).

Resposta simples: Se você não configurou zonas, é como se o prédio não tivesse segurança nenhuma. O ZPF só entra em ação quando você define zonas. Por isso, a primeira etapa de configuração é criar as zonas e atribuí-las às interfaces.

Resposta direta: Nada passa! A regra padrão do ZPF é "bloquear tudo que não for explicitamente permitido". É como ter um prédio onde todas as portas estão trancadas até você destrancar especificamente as que precisam ficar abertas.

Essa é uma abordagem de segurança muito mais segura do que "permitir tudo e bloquear o que é perigoso".
