3.1.1 IPv4 e IPv6

O IP foi projetado como um protocolo sem conexão de Camada 3. Ele fornece as funções necessárias para entregar um pacote de um host de origem a um host de destino por meio de um sistema interconectado de redes. O protocolo não foi projetado para rastrear e gerenciar o fluxo de pacotes. Essas funções, se necessárias, são executadas principalmente pelo TCP na camada 4.

Vulnerabilidade Crítica:

O IP não faz nenhum esforço para validar se o endereço IP de origem contido em um pacote realmente veio dessa origem.

Impacto:

Agentes de ameaças podem enviar pacotes usando endereço IP de origem falsificado (spoofing) e adulterar outros campos do cabeçalho IP para realizar seus ataques.

PARA ANALISTAS DE SEGURANÇA:

É importante que os analistas de segurança entendam os diferentes campos dos cabeçalhos IPv4 e IPv6 para identificar ataques como spoofing, fragmentação maliciosa e exfiltração de dados.

💣 Caso de Uso: Ataque de Opções IP (1990s) - Roteadores antigos processavam opções IP em software (não em hardware), causando sobrecarga de CPU. Atacantes enviavam pacotes com múltiplas opções para degradar performance do roteador.

3.2.2 Ataques ICMP

O ICMP foi desenvolvido para transportar mensagens de diagnóstico e relatar condições de erro quando rotas, hostes e portas não estão disponíveis. Mensagens ICMP são geradas por dispositivos quando ocorre um erro de rede ou uma interrupção. O comando ping é uma mensagem ICMP gerada pelo usuário, chamada de solicitação de eco, usada para verificar a conectividade com um destino.

Uso Malicioso:

Os agentes de ameaças usam o ICMP para ataques de reconhecimento e varredura. Isso permite que eles lançem ataques de coleta de informações para mapear uma topologia de rede, descobrir quais hosts estão ativos (acessíveis), identificar o sistema operacional do host (impressão digital do sistema operacional) e determinar o estado de um firewall.

Ataques DoS/DDoS:

Os atores de ameaças também usam ICMP para ataques DoS e DDoS, conforme mostrado no ataque de inundação ICMP.

Recomendações para Analistas de Segurança:

As redes devem ter uma filtragem rigorosa da lista de controle de acesso ICMP (ACL) na borda da rede para evitar a sondagem ICMP vindo da Internet.

Os analistas de segurança devem ser capazes de detectar ataques relacionados ao ICMP, observando o tráfego capturado e os arquivos de log. No caso de grandes redes, os dispositivos de segurança, como firewalls e sistemas de detecção de intrusão (IDS), devem detectar tais ataques e gerar alertas para os analistas de segurança.

3.2.3 Ataques de negação de serviço (DoS)

Os atores da ameaça tentam impedir que usuários legítimos acessem informações ou serviços.

🔍 Exemplo: Ataque SYN Flood - invasor envia múltiplas solicitações de conexão TCP com endereços de origem falsificados, esgotando a capacidade do servidor de aceitar novas conexões legítimas.

1. ICMP requisição Echo (echo request) origem = IP da vítima (falsificado)

1. Amplification

The threat actor forwards ICMP echo request messages to many hosts. Essas mensagens contêm o endereço IP de origem da vítima.

2. Reflection

These hosts all reply to the spoofed IP address of the victim with overwhelming traffic.

💣 Caso Real: Ataque Smurf (1998)

O ataque Smurf é um exemplo clássico de amplificação/reflexão. O atacante envia pacotes ICMP echo request para o endereço de broadcast de uma rede, com endereço de origem falsificado da vítima. Todos os hosts na rede respondem, gerando tráfego massivo direcionado à vítima.

3.3.2 Serviços TCP e Handshake Triplo

🔧 Serviços fornecidos pelo TCP

  • Entrega confiável - Reconhecimentos (ACKs) e retransmissão (HTTP, SSL/TLS, FTP, DNS zone transfers)
  • Controle de fluxo - Janela deslizante para otimizar transmissão
  • Comunicação stateful - Conexão estabelecida via handshake triplo

🔄 Aperto de mão de três vias TCP

Cliente                          Servidor
   |                                |
   |-------- SYN (SEQ=100) -------->|
   |                                |
   |<----- SYN-ACK (SEQ=300, ACK=101)|
   |                                |
   |-------- ACK (SEQ=101, ACK=301)->|
   |                                |
   |<==== CONEXÃO ESTABELECIDA ====>|
  1. O cliente iniciador requisita uma sessão (SYN)
  2. O servidor confirma e requisita sessão reversa (SYN-ACK)
  3. O cliente confirma (ACK) e a conexão é estabelecida
APLICAÇÃO PRÁTICA:

Firewalls e sistemas de detecção de intrusão (IDS) monitoram o handshake TCP para identificar varreduras de porta furtivas (Stealth Scan) e conexões incompletas que podem indicar atividades maliciosas.

3.3.3 Ataques TCP

Ataque de Inundação SYN (SYN Flood)

O ataque de inundação SYN explora o handshake triplo do TCP. O agente de ameaça envia continuamente pacotes SYN com endereços IP de origem falsificados. O servidor responde com SYN-ACK e aguarda o ACK final, que nunca chega, consumindo recursos e negando serviço a usuários legítimos.

         |---- SYN (IP real) ----------->|
         |<--- SYN-ACK -------------------|
         | (servidor sem recursos p/ responder) |
  1. Ator da ameaça envia múltiplas solicitações SYN com IPs falsificados
  2. Servidor responde com SYN-ACK e mantém conexões semi-abertas
  3. Usuário legítimo não consegue acessar devido ao esgotamento de recursos
📌 CASO REAL:

Em 2016, o provedor DNS Dyn sofreu um ataque DDoS massivo que utilizou SYN floods entre outras técnicas, derrubando serviços como Twitter, Netflix e Spotify. Mitigações incluem SYN cookies e rate-limiting implementados em firewalls e balanceadores de carga.

Ataque de Redefinição TCP (TCP RST Attack)

O TCP utiliza uma troca de quatro vias (FIN/ACK) para encerrar conexões normalmente, mas o flag RST encerra abruptamente. Um agente de ameaça pode enviar um pacote falsificado com RST para uma ou ambas as partes, forçando o encerramento imediato da conexão. 

Encerramento Normal (4 vias)          Ataque RST
   A               B                     A               B
   |               |                     |               |
   |---- FIN ----->|                     |               |
   |<--- ACK ------|                     |               |
   |               |                     |<==== RST =====|
   |<--- FIN ------|                     | (conexão      |
   |---- ACK ----->|                     |  encerrada)   |
   |               |                     |               |
📌 CASO REAL:

Ataques RST são usados em censura de rede e interrupção de conexões P2P. Em 2011, foi descoberto que um ISP utilizava pacotes RST forjados para interromper conexões BitTorrent de seus usuários. Ferramentas como TCP Reset Attack Tool automatizam esse tipo de ataque.

Sequestro de Sessão TCP (TCP Session Hijacking)

O sequestro de sessão TCP é uma vulnerabilidade avançada onde o agente assume um host já autenticado. O atacante deve falsificar o IP de um dos hosts, prever o próximo número de sequência e enviar um ACK para o outro host. Se bem-sucedido, pode enviar dados (mas não receber) como se fosse o host legítimo.

📌 CASO REAL:

O ataque de Kevin Mitnick em 1994 contra o computador de Tsutomu Shimomura no San Diego Supercomputer Center utilizou sequestro de sessão TCP combinado com predição de números de sequência. Este ataque histórico levou ao desenvolvimento de mecanismos como números de sequência aleatórios e criptografia (IPSec, SSL/TLS) para proteger sessões TCP.

3.3.4 Cabeçalho e Operação do Segmento UDP

UDP é comumente usado por DNS, DHCP, TFTP, NFS e SNMP. Também é utilizado em aplicações em tempo real como streaming de mídia ou VoIP. UDP é um protocolo não orientado à conexão, com sobrecarga muito menor que TCP, pois não oferece retransmissão, sequenciamento ou controle de fluxo.

Nota: UDP divide dados em datagramas, mas o termo genérico "segmento" é comumente usado.

📦 Estrutura do Segmento UDP (8 bytes)

Bit (0) Bit (15) Bit (16) Bit (31)
Porta de Origem (16) Porta de Destino (16)
Comprimento (16) Checksum (16)
Dados da Camada de Aplicação (tamanho variável)

Embora UDP seja chamado de "não confiável", isso não significa que aplicações que o utilizam são inferiores. A baixa sobrecarga o torna ideal para transações simples como DHCP: se não há resposta, o dispositivo reenvia a solicitação.

📌 CASO REAL:

DNS (Domain Name System) utiliza UDP na porta 53 para consultas rápidas. Ataques de amplificação DNS exploram isso: atacantes enviam consultas com IP falsificado (da vítima) para servidores DNS abertos, que respondem com pacotes muito maiores, amplificando o tráfego contra a vítima.

3.3.5 Ataques UDP

Falta de Criptografia no UDP

O UDP não está protegido por nenhuma criptografia nativa. Embora seja possível adicionar criptografia (como DTLS), por padrão qualquer pessoa pode ver o tráfego, alterá-lo e enviá-lo ao destino. O checksum de 16 bits é opcional e, quando usado, um atacante pode recalcular e modificar o payload sem que o destino perceba. 

Pacote UDP original:
[PORTA ORIGEM] [PORTA DESTINO] [COMP] [CHECKSUM] [DADOS: "saldo=1000"]

Ataque de manipulação:
[PORTA ORIGEM] [PORTA DESTINO] [COMP] [NOVO CHECKSUM] [DADOS: "saldo=0000"]

→ O destino valida o checksum (agora correto) e aceita os dados alterados.
📌 CASO REAL:

TFTP (Trivial File Transfer Protocol) utiliza UDP sem criptografia. Em redes internas, atacantes podem interceptar e modificar arquivos de configuração transferidos para switches e roteadores, instalando backdoors ou causando falhas. A solução é utilizar TFTP com IPsec ou migrar para SFTP/SCP.

Ataque de Inundação UDP (UDP Flood)

Em um ataque de inundação UDP, todos os recursos de uma rede são consumidos. O agente utiliza ferramentas como UDP Unicorn ou Low Orbit Ion Cannon (LOIC) para enviar uma enxurrada de pacotes UDP (geralmente com IP falsificado) para um servidor na sub-rede. O programa varre todas as portas conhecidas buscando portas fechadas, fazendo o servidor responder com mensagens ICMP "porta inacessível", consumindo largura de banda. 

ATACANTE (IP Falsificado)         SERVIDOR ALVO
         |                                |
         |---- UDP porta 1 (falso) ------>|
         |---- UDP porta 2 (falso) ------>|
         |---- UDP porta 3 (falso) ------>|
         |              ...               |
         |                                |
         |<--- ICMP Port Unreachable (p1)--|
         |<--- ICMP Port Unreachable (p2)--|
         |<--- ICMP Port Unreachable (p3)--|
         |              ...               |
         |                                |
         | (largura de banda saturada)    |
         |                                |
         USUÁRIO LEGÍTIMO                 |
         |---- UDP porta válida -------->|
         | (sem resposta - rede congestionada)|
  1. Atacante envia pacotes UDP para portas aleatórias do alvo
  2. Servidor responde com ICMP "Destination Unreachable" para portas fechadas
  3. O grande volume de respostas consome largura de banda e recursos
  4. Resultado similar a um ataque de negação de serviço (DoS)
📌 CASO REAL:

Em 2018, o provedor de jogos online Steam sofreu ataques UDP flood direcionados a servidores de jogos populares. Ferramentas como LOIC foram utilizadas por grupos hacktivistas. Mitigações incluem rate-limiting de respostas ICMP, firewalls stateful e serviços de proteção contra DDoS como Cloudflare e AWS Shield.

3.4 Atacando a Fundação - Resumo

Clique nos cabeçalhos abaixo para ver um resumo dos tópicos cobertos neste módulo.

O IP foi projetado como um protocolo sem conexão de Camada 3. O cabeçalho IPv4 consiste em vários campos, enquanto o cabeçalho IPv6 contém menos campos. É importante que os analistas de segurança entendam os diferentes campos dos cabeçalhos IPv4 e IPv6.

IPv4
  • • Versão (0100)
  • • IHL (20-60 bytes)
  • • Serviços Diferenciados
  • • TTL e Protocolo
  • • Endereços Origem/Destino
IPv6
  • • Versão (0110)
  • • Classe de tráfego
  • • Rótulo de fluxo
  • • Tamanho da carga
  • • Próximo cabeçalho
  • • Limite de saltos
🔍 Principais vulnerabilidades: IP Spoofing, Fragmentação maliciosa, Opções IP maliciosas.

Existem diferentes tipos de ataques que visam IP. Os ataques comuns relacionados a IP incluem:

Ataques ICMP

O ICMP foi desenvolvido para transportar mensagens de diagnóstico e relatar condições de erro. Os agentes de ameaças usam o ICMP para ataques de reconhecimento, varredura, DoS e DDoS.

DoS e DDoS

Ataques de negação de serviço e distribuídos. Os agentes de ameaças geralmente usam técnicas de amplificação e reflexão para criar ataques de negação de serviço (DoS).

Amplificação/Reflexão

Técnicas usadas para amplificar ataques DoS, como o ataque Smurf e amplificação de DNS.

Falsificação de Endereços

Ocorre quando um agente de ameaça cria pacotes com informações falsas de endereço IP de origem para ocultar a identidade do remetente ou se passar por outro usuário legítimo.

  • • Não cega: para sequestrar sessão
  • • Cega: para criar ataques DoS
MiTM e Sequestro

Ataque man-in-the-middle e sequestro de sessão. Os ataques de falsificação de endereço MAC são usados quando os atores de ameaças têm acesso à rede interna.

Exaustão de Recursos

Ataques que consomem os recursos de um host de destino para travá-lo ou consomem os recursos de uma rede.

🔍 Resumo: O ICMP é usado para reconhecimento e ataques. Spoofing pode ser cego ou não cego. MiTM e sequestro de sessão são ataques avançados que combinam múltiplas técnicas.

As informações do segmento TCP e do datagrama UDP aparecem imediatamente após o cabeçalho IP. É importante entender cabeçalhos da Camada 4 e suas funções na comunicação de dados.

TCP

O TCP fornece entrega confiável, controle de fluxo e comunicação com estado. A comunicação com monitoração de estado TCP entre duas partes ocorre durante o handshake TCP de três vias.

Ataques relacionados ao TCP:
  • • Varreduras de porta TCP
  • • Ataque de Inundação de SYN TCP
  • • Ataque de redefinição de TCP (TCP Reset)
  • • Ataque de sequestro de sessão TCP
💣 SYN Flood: Atacante envia múltiplos SYN com IPs falsificados, esgotando a fila de conexões do servidor.
UDP

O segmento UDP (datagrama) é muito menor do que o segmento TCP, o que o torna muito desejável para uso por protocolos que fazem transações simples de solicitação e resposta, como DNS, DHCP, SNMP e outros.

Ataques relacionados ao UDP:
  • • Ataques de inundação UDP
  • • Varredura de portas UDP conhecidas
  • • Amplificação DNS (UDP)
  • • Ataques a serviços UDP vulneráveis

Os atores de ameaças podem conduzir ataques de inundação UDP que varrem todas as portas UDP conhecidas em um servidor tentando encontrar portas fechadas. Isso pode criar uma situação DoS.

💣 UDP Flood: Atacante envia grandes volumes de datagramas UDP para portas aleatórias, forçando o servidor a verificar serviços inexistentes.
